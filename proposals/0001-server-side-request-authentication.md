# Proposal 0001: Server-Side Request Authentication (proxy-delegated identity)

- Issue: #19
- Status: Proposed
- Date: 2026-06-05

## Summary

A deployment may run the registry behind a gateway that has already authenticated the caller (OIDC ingress, OAuth2 proxy, identity-verifying sidecar, or a non-OIDC corporate SSO) and wants per-layer visibility filtering (§4.6) driven by that identity. The registry-side identity provider (`PODIUM_IDENTITY_PROVIDER`, §6.3) covers `oauth-device-code` and `injected-session-token` today, but it cannot verify a user's OIDC token forwarded by a gateway, nor trust gateway-supplied identity headers.

This proposal adds the registry-process values `oidc-jwt` and `trusted-headers` to the existing `PODIUM_IDENTITY_PROVIDER` selector rather than a parallel setting: `oidc-jwt` verifies a forwarded token, and `trusted-headers` trusts gateway-injected headers. The registry process reads these values; the MCP server's value-set is unchanged. The registry does not yet ship a server-side OIDC verifier or a header-identity resolver, so this proposal introduces the server-side verification and identity-resolution path that derives the caller identity (`sub`, `email`, and groups) and filters layer visibility by that identity (§4.6), reusing the existing `PODIUM_OAUTH_AUDIENCE` configuration and the §6.3.1 claim-derivation contract. The step that selects the §4.7.1 tenant from a verified `org_id` claim or the `X-Podium-User-Org` header is a new mechanism this proposal introduces; §6.3.1 does not describe per-request `org_id`-to-tenant routing today, and the registry core does not yet implement it (§13.1). `oidc-jwt` reports an expired token with `auth.token_expired`, rejects a forwarded token that fails signature, `iss`, or `aud` validation with `auth.untrusted_token`, and rejects a verified token whose `org_id` names no provisioned tenant on a registry running in multi-tenant mode with `auth.tenant_unknown`. The spec amendment below documents these codes: `auth.token_expired` for the `injected-session-token` expiry case that the implementation already emits but the spec leaves unnamed, extended to `oidc-jwt`, `auth.untrusted_token` for a forwarded token that fails verification, and `auth.tenant_unknown` for a verified token whose `org_id` names no provisioned tenant. The existing `auth.untrusted_runtime` code stays specific to the §6.3.2 runtime case and is not reused. `oidc-jwt` adds an OIDC issuer setting to the `identity_provider:` block, `issuer` (with `PODIUM_OAUTH_ISSUER` as the environment override), from which the registry derives the JWKS. They add `PODIUM_OAUTH_TOKEN_HEADER`, `PODIUM_OAUTH_JWKS_CACHE_TTL_SECONDS`, and `PODIUM_TRUSTED_PROXY_SECRET`. They apply to any server deployment fronted by a gateway, on a standalone or a standard backend, and are mutually exclusive with public mode. The sections below are drafted for `spec/06-mcp-server.md` and `spec/13-deployment.md`.

## 6.3.3 Server-Side Request Authentication

The registry resolves the §4.6 caller identity from each request through the provider selected by `PODIUM_IDENTITY_PROVIDER` (§6.3). Alongside `oauth-device-code` and `injected-session-token`, two values support a gateway that has authenticated the caller before the request reaches the registry. `oidc-jwt` and `trusted-headers` are registry-process values, selected by the registry's `PODIUM_IDENTITY_PROVIDER`. They configure how the registry resolves identity from each request; a Podium client (CLI, MCP server) acquires and sends no credential for them, because identity is supplied by the gateway. These two values are not MCP-server (client) provider selections: the MCP server's `PODIUM_IDENTITY_PROVIDER` admits only the client-side selections `oauth-device-code` and `injected-session-token` (§6.3), and rejects any other value at startup (`cmd/podium-mcp/main.go`). A deployment sets `oidc-jwt` or `trusted-headers` on the registry process (`podium-server` / `podium serve`), and leaves the MCP server on its client-side default. Whether a client can reach the registry directly without losing its identity depends on the mode, and each value states its own network-path requirement below. `oidc-jwt` verifies the forwarded token cryptographically on every request, so the registry may be directly reachable and a token presented outside the gateway still resolves to its subject. `trusted-headers` reads identity from gateway-injected headers it cannot verify, so it rests on the operational assumption that every request arrived through the gateway, and a request that bypasses the gateway carries no trusted identity and the caller sees public visibility only (§4.6).

Both apply to any server deployment whose registry is fronted by such a gateway, on a standalone (§13.10) or a standard (§13.1) backend. They are not part of the zero-config standalone default, which stays unauthenticated and loopback-bound (§13.10); enabling one opts a server into gateway-delegated identity, and on a standalone backend it relaxes the §13.11.6 trigger to migrate to a remote server for identity-based filtering. Both are mutually exclusive with public mode: setting either alongside `PODIUM_PUBLIC_MODE` fails at startup with `config.public_mode_with_idp`, as for any identity provider (§13.10). Both record the subject and email and match them as §4.6 `users:` permits, and both derive the caller's organization (the §4.7.1 tenant) from the authenticated identity rather than a value the caller's client supplies. When the registry runs in multi-tenant mode (more than one provisioned tenant, or an explicit operator opt-in per the §6.3.3 startup signal), this proposal specifies that `oidc-jwt` selects the tenant from the verified `org_id` claim and `trusted-headers` from the gateway-set `X-Podium-User-Org` header. Multi-tenant provisioning is available on the standard backend (§13.1); a standard deployment booting the single `default` tenant is single-tenant. Per-request `org_id`-to-tenant routing is a new mechanism the §6.3.1 amendment below defines, and it depends on the registry-core tenant-routing substrate the §6.3.1 amendment below lists as a prerequisite (the boot-time `core.Registry` tenant, `core.New(st, tenantID, layers)` in `internal/serverboot/serverboot.go` and `pkg/registry/server/server.go`, which today selects the single bootstrapped tenant per `internal/serverboot/orgid.go`). A registry running single-tenant, whether a standalone backend (§13.10) or a standard backend with only the `default` tenant, resolves every authenticated caller to its sole tenant without consulting that organization value. They differ in how groups are resolved, as each value describes below.

**Implementation prerequisite (fail-closed registration).** `oidc-jwt` and `trusted-headers` do not exist in the current binary: there is no server-side token verifier or header-identity resolver, and `identity.Default` registers only `oauth-device-code` and `injected-session-token` (`pkg/identity/registry.go`). Because they are unregistered, `selectIdentityProvider` returns no provider, `identityVisibilityGuard` treats them as the externally-fronted exempt case (`!providerSelected`) and does not refuse startup, and the registry falls back to the anonymous-public resolver (`server.New`). Selecting either value on a build that predates this implementation therefore starts a registry that resolves every caller as anonymous-public and applies no per-layer visibility, with no verifier, no error, and no startup warning. The implementation must make both values fail closed before release: register them so `providerSelected` becomes true and a missing verifier or resolver is rejected by `identityVisibilityGuard` (`config.identity_provider_unverified`), or extend `identityVisibilityGuard` so `oidc-jwt` and `trusted-headers` are not covered by the `!providerSelected` exemption. This is one of three prerequisites this proposal depends on, alongside the per-request `org_id`-routing prerequisite the §6.3.1 amendment below states and the multi-tenant-mode startup signal stated immediately below, so a phased rollout cannot open a registry.

**Multi-tenant-mode startup signal.** The bind rules and startup codes below that key on tenancy require a concrete startup signal that places the registry in multi-tenant mode, and this proposal defines that signal as a third prerequisite. The registry is in multi-tenant mode when it has more than the single bootstrapped `default` tenant provisioned, through an org-provisioning path beyond `bootstrapDefaultTenant` (`internal/serverboot/serverboot.go:700`), or when the operator opts in explicitly (for example a `PODIUM_MULTI_TENANT` flag). Neither the org-provisioning path nor that opt-in exists in the current binary, so this signal is itself unimplemented and lands with the per-request routing substrate. `storeType == postgres` alone is not the signal: `bootstrapDefaultTenant` seeds the single `default` org unconditionally for every store type (`serverboot.go:700`), and `core.New(st, tenantID, layers)` binds one `tenantID` for the process lifetime, so a Postgres/§13.1 deployment that has only the `default` org is single-tenant. A §13.1 standard deployment is therefore not synonymous with a multi-tenant registry; the standard topology (§13.1) describes the Postgres-backed stack, and §4.7.1 defines the org as the tenant boundary, but neither provisions a second org on its own. The bind rules below trigger on multi-tenant mode under this definition rather than on the standard backend.

`PODIUM_IDENTITY_PROVIDER` selects one provider per registry process, and that provider fixes the single trust root the registry verifies against on every request: a JWKS for `oidc-jwt`, registered runtime signing keys for `injected-session-token`, and gateway-injected headers for `trusted-headers`. This proposal adds two values to that selector and does not add a combined value or a list of providers, so a single registry process cannot run two providers at once. The constraint is not specific to the new values: `oauth-device-code` and `injected-session-token` already cannot coexist in one process for the same reason.

A registry that must admit both gateway-authenticated users and managed runtimes therefore selects one trust model. Where both populations reach the registry through the same gateway, one `oidc-jwt` provider covers both: a runtime fronted by that gateway forwards an `oidc-jwt` token like any other caller, and its forwarded token verifies against the same issuer JWKS. Where the two populations require distinct trust roots in one deployment (for example gateway-delegated `oidc-jwt` for users and runtime-signature `injected-session-token` for runtimes whose tokens carry an `iss` that names a registered runtime, §6.3.2), the deployment runs two registry processes against the same backend, each with its own `PODIUM_IDENTITY_PROVIDER`, and routes each population to the matching endpoint. The gateway-delegated values are for a deployment that resolves user identity at the gateway.

**`oidc-jwt` (verified).** The gateway forwards the caller's IdP-signed JWT in the header named by `token_header` (default `Authorization`). The registry parses the named header's value as the standard HTTP Bearer credential regardless of the header name: the value must be `Bearer <token>`, where `<token>` is the JWT. The same parsing applies to a custom `token_header` (for example `X-Podium-Forwarded-JWT`), so the gateway writes `Bearer <token>` into whichever header it forwards. The registry matches the `Bearer ` prefix case-insensitively and trims surrounding whitespace from `<token>` before verification. A header value that lacks the `Bearer ` prefix carries no token, so the request is anonymous. The registry verifies the token on every request: it selects the signing key by `kid` from the issuer's JWKS (resolved from the configured `issuer` URL through its OIDC discovery document at `${issuer}/.well-known/openid-configuration` and refreshed when the cached key set is older than `PODIUM_OAUTH_JWKS_CACHE_TTL_SECONDS`, default 300), checks the signature, and validates `iss` against the configured `issuer`, `aud` against `PODIUM_OAUTH_AUDIENCE`, and the `exp`/`nbf` window. On success it records `sub` and `email`, derives the caller's organization (the §4.7.1 tenant) from the verified `org_id` claim under the routing the §6.3.1 amendment below introduces, and resolves groups through the §6.3.1 `IdpGroupMapping` adapter applied to the token's group claim; on a standard (§13.1) backend it can also resolve groups through SCIM (§6.3.1). SCIM-based resolution is unavailable on a standalone (§13.10) backend, where §13.10 scopes SCIM out, so a standalone `oidc-jwt` server resolves groups through `IdpGroupMapping` alone. On a standard (§13.1) backend the `org_id` claim carries an org ID or an org-name alias (§4.7.1), and the §6.3.1 amendment below specifies that the registry resolves the alias to its org ID and selects the tenant whose layer list (§4.6) and audit stream the request resolves against, so a multi-tenant registry routes each verified token to its org without a caller-supplied tenant value. That per-request routing depends on the registry-core tenant-routing substrate the §6.3.1 amendment below lists as a prerequisite (the boot-time `core.Registry` tenant, `core.New(st, tenantID, layers)` in `internal/serverboot/serverboot.go` and `pkg/registry/server/server.go`, which today selects the single bootstrapped tenant per `internal/serverboot/orgid.go`). On a standalone (§13.10) backend the registry stays single-tenant: §13.10 scopes multi-tenancy out of standalone, the registry seeds a single tenant at boot (the synthetic `default` org, whose ID is a name-derived UUID), and enabling `oidc-jwt` does not enable multi-tenant routing. There the registry resolves every verified request against that sole tenant and does not consult `org_id` for tenant selection: the seeded org is synthetic, so an external IdP's `org_id` would not name it, and requiring a match would reject otherwise-valid tokens. On a standalone backend the registry maps every verified caller to its sole tenant without consulting `org_id`, so the configured `PODIUM_OAUTH_AUDIENCE` is the only organization boundary. Any subject for whom the fronting IdP issues a token bearing this audience is treated as an authenticated member of the standalone org. Such a caller can therefore read every `organization: true` layer (§4.6) and any `groups:`/`users:` layer whose listed values match the token's claims, even if the subject belongs to a different organization at the IdP. Layers marked `private` with an explicit `users:` list, or `groups:`, are reachable only by callers whose claims match those values. When the fronting IdP issues the same audience to more than one organization (for example Auth0 Organizations, an Entra multi-tenant application, or an Okta resource audience shared across orgs), scope `PODIUM_OAUTH_AUDIENCE` to this server alone and configure the IdP or gateway to issue that audience only to the intended population. Multi-tenant routing by verified `org_id` requires a standard backend (§13.1). A token that fails signature, `iss`, or `aud` validation is rejected with `auth.untrusted_token`; on a registry running in multi-tenant mode, a verified token whose `org_id` names no provisioned tenant is rejected with `auth.tenant_unknown`. An expired token is rejected with `auth.token_expired`. The spec amendment below defines these codes for §6.9 and §6.10. A request carrying no token is anonymous and sees public visibility only (§4.6). This mode trusts the issuer's signing key alone and trusts no element of the network path, so the registry may be directly reachable without an authentication bypass. It applies the §6.3.2 signature-verification model, a signature checked against a trusted key, with the key taken from the issuer's JWKS rather than a registered runtime key. The configured `issuer` must use the `https` scheme, because the registry fetches the OIDC discovery document and the JWKS over that URL and an `http` endpoint lets a man-in-the-middle substitute a signing key; the registry rejects a non-`https` `issuer` at startup with `config.invalid_issuer_scheme` and does not start. If `PODIUM_OAUTH_AUDIENCE` is unset, the registry rejects startup with `config.oidc_jwt_audience_unset` and does not start, because an unverifiable `aud` would accept a token issued for any relying party that shares the issuer. If the IdP's OIDC discovery document or JWKS, resolved from the configured `issuer`, is unreachable at startup, the registry fails to start; while the key set is unavailable at runtime, verification fails closed and the request is anonymous; a `kid` absent from the cached set triggers a refresh before rejection.

**`trusted-headers` (delegated).** This mode rests on one operational assumption that the registry cannot verify: every request arrived through the gateway, and the gateway removed any client-supplied `X-Podium-User-*` headers before setting its own. An operator who enables it owns that assumption and the controls (a proxy secret, a bind restriction, or an upstream network control) that hold it true. The gateway authenticates the caller by any means and injects the resolved identity as request headers: `X-Podium-User-Sub`, `X-Podium-User-Email`, `X-Podium-User-Groups` (comma-separated), and `X-Podium-User-Org`. The registry records `sub` and `email` from these headers and performs no verification of their contents. Groups come from `X-Podium-User-Groups` directly, because there is no token for the §6.3.1 `IdpGroupMapping` adapter to read and the gateway is the source of truth; SCIM is not consulted in this mode. The caller's organization (the §4.7.1 tenant) comes from `X-Podium-User-Org`, which carries an org ID or an org-name alias (§4.7.1). On a standard (§13.1) backend this proposal specifies that the header selects the tenant whose layer list (§4.6) and audit stream the request resolves against, so a multi-tenant registry routes each request to its org by this value. That per-request routing is the new mechanism the §6.3.1 amendment below defines, and it depends on the registry-core tenant-routing substrate the §6.3.1 amendment below lists as a prerequisite (the boot-time `core.Registry` tenant, `core.New(st, tenantID, layers)` in `internal/serverboot/serverboot.go` and `pkg/registry/server/server.go`, which today selects the single bootstrapped tenant per `internal/serverboot/orgid.go`). The registry trusts this header as it trusts the others, because the gateway is the source of truth and no token carries the claim. On a standalone (§13.10) backend the registry stays single-tenant: §13.10 scopes multi-tenancy out of standalone, the registry seeds a single tenant at boot, and enabling `trusted-headers` does not enable multi-tenant routing. There the registry resolves every request against the sole tenant and does not consult `X-Podium-User-Org` for tenant selection, because the seeded org is synthetic and an external org value would not name it. On a multi-tenant (standard) registry, `X-Podium-User-Org` selects the tenant: a request that omits it is anonymous and sees public visibility only, because the registry cannot place it in a tenant, and an `X-Podium-User-Org` that names an unknown org is treated the same way. Because the registry performs no verification, it trusts the network path: the bind restriction and proxy-secret controls below are how the operator backs the assumption stated at the top of this mode. A request without identity headers is anonymous and sees public visibility only.

When `PODIUM_TRUSTED_PROXY_SECRET` is set, the registry honors the `X-Podium-User-*` headers only on a request whose `X-Podium-Proxy-Secret` header matches the configured value under a constant-time comparison; on any other request it discards the identity headers and treats the caller as anonymous. When the secret is unset, `X-Podium-Proxy-Secret` is ignored and the identity headers are honored on every request.

Because `trusted-headers` does not verify the headers, the identity it trusts is exactly the set of clients that can reach the bind address, so the mode constrains the bind at startup (`oidc-jwt` verifies every token and carries no such restriction). The bind rule keys on multi-tenant mode under the startup signal defined above (more than one provisioned tenant, or an explicit operator opt-in) rather than on the standard backend (§13.1), so a §13.1 standard deployment booting the single `default` tenant is single-tenant and falls under the single-tenant rule. On a registry running single-tenant, a loopback bind (`127.0.0.0/8`, `::1`) is allowed without a proxy secret, because there is no caller-supplied tenant selection: the registry resolves every request to its sole synthetic tenant and does not consult `X-Podium-User-Org`, so a co-resident process that forges identity headers reaches the same single tenant the operator already owns. On a registry running in multi-tenant mode, where `X-Podium-User-Org` selects among tenants, the loopback exemption does not apply: a loopback bind in a shared-network-namespace pod (the sidecar-gateway pattern) is reachable by every co-resident container on `127.0.0.1`, so co-residency does not authenticate the gateway, and with unverified `X-Podium-User-Org` tenant selection a co-resident process could read any tenant by forging the header. A registry running in multi-tenant mode under `trusted-headers` therefore requires `PODIUM_TRUSTED_PROXY_SECRET` regardless of bind address, including loopback, as a per-request constant-time gate on header trust; when it is unset the registry refuses to start with `config.trusted_headers_multitenant_no_secret`. On a registry running single-tenant, including a Postgres/§13.1 deployment that has only the `default` tenant, a non-loopback bind fails to start with `config.trusted_headers_public_bind` unless a proxy secret (`PODIUM_TRUSTED_PROXY_SECRET`) or `--allow-public-bind` is configured; the `--allow-public-bind` escape does not apply in multi-tenant mode, where the secret is required regardless of bind address per the preceding sentence. The spec amendment below ("§13.10 Bind restriction under `trusted-headers`") defines this startup check as a §13.10 subsection scoped to the `trusted-headers` provider, separate from the public-mode "Loopback bind by default" rule, and names the error code. The proxy secret is the registry's only request-level control over header trust, because the registry serves HTTP and cannot itself require or verify a client certificate (TLS terminates upstream at the gateway or load balancer); a direct caller that lacks the secret is treated as anonymous. `--allow-public-bind` carries no such control: it records the operator's explicit assumption that an upstream property the registry cannot verify, such as mutual TLS at the load balancer or mesh, a firewall, or a network policy, keeps the registry reachable only through the gateway, and with it (and no secret) the registry trusts the headers on every request. A non-loopback bind with neither refuses to start rather than serve forgeable identity by default. This bind check runs at startup and complements the per-request secret comparison above. The registry prints a startup banner stating that caller identity is delegated to the upstream gateway.

Prefer `oidc-jwt` where the gateway can forward a verifiable token. Use `trusted-headers` only where it cannot, such as a gateway that authenticates via SAML or a bespoke SSO and forwards identity as headers alone, and pair it with a proxy secret or an upstream control over who can reach the registry.

The `X-Podium-User-*` and `X-Podium-Proxy-Secret` headers are request inputs read only in `trusted-headers` mode. They are unrelated to the §13.2.1 `X-Podium-*` response headers (a different direction), the registry sets none of them on a response, and `trusted-headers` does not consult the `X-Forwarded-User` header as an authorization input. `X-Forwarded-User` is the public-mode audit annotation that the registry records in `caller.network` (§8.1, §13.10); a `trusted-headers` deployment is mutually exclusive with public mode, so its authenticated read events record `caller.email` and `caller.groups` rather than `caller.network`, and `X-Forwarded-User` is not consulted for identity in this mode.

## 6.10 (additions)

`oidc-jwt` rejects a forwarded token that fails signature, `iss`, or `aud` validation with `auth.untrusted_token`, a new code defined by the spec amendment below; on a registry running in multi-tenant mode it rejects a verified token whose `org_id` names no provisioned tenant with `auth.tenant_unknown`, because the token passed verification and the failure is tenancy, so the verification-failure message and remediation do not apply. It reports an expired token with `auth.token_expired`, the code the spec amendment below documents for the existing `injected-session-token` expiry case and extends to `oidc-jwt`. `auth.untrusted_token` is distinct from `auth.untrusted_runtime`: the runtime code (§6.3.2) names a runtime whose signing key is not registered and directs the operator to `podium admin runtime register`, while the token code names the forwarded token's issuer and directs the operator to the `oidc-jwt` issuer and audience configuration. `trusted-headers` raises no authentication error: a missing or unauthenticated identity yields anonymous, public-only visibility rather than a rejection.

## Spec amendment: §6.3.1 per-request `org_id`-to-tenant routing

This subsection drafts the spec change for `spec/06-mcp-server.md` and depends on a registry-core prerequisite stated at the end. §4.7.1 establishes the tenant boundary as the org, gives each org its own layer list (§4.6), admins, audit stream, and quotas, and defines org IDs as UUIDs with org names as human-readable aliases. Neither §6.3.1 nor §4.7.1 defines how a request on a registry running in multi-tenant mode selects its tenant from an authenticated identity. The gateway-delegated providers require that selection, so this proposal adds it as a new mechanism rather than reusing the §6.3.1 claim-derivation path, which derives `sub`, `email`, and groups but not the tenant.

Add to §6.3.1 the per-request tenant-selection rule for a registry running in multi-tenant mode. Multi-tenant provisioning is available on the standard backend (§13.1), and the rule applies when more than one tenant is provisioned (or the operator opts in explicitly per the §6.3.3 startup signal). A standard deployment booting the single `default` tenant is single-tenant and falls under the single-tenant rule below.

> When the registry is running in multi-tenant mode, it selects the request's tenant from the organization value carried by the authenticated identity: the verified `org_id` claim under `oidc-jwt`, or the `X-Podium-User-Org` header under `trusted-headers`. The value is an org ID or an org-name alias (§4.7.1); the registry resolves an alias to its org ID and selects that org's layer list (§4.6) and audit stream for the request. A value that resolves to no provisioned tenant is rejected with `auth.tenant_unknown` (§6.10) under `oidc-jwt`; under `trusted-headers` the request is treated as anonymous and sees public visibility only (§4.6). A registry running single-tenant, including a single-tenant standalone backend (§13.10) and a standard backend (§13.1) that has only the `default` tenant, resolves every authenticated request against its sole tenant and does not consult the organization value.

This per-request `org_id`-to-tenant routing is a prerequisite this proposal depends on and the registry core does not yet implement. The core binds a single boot-time tenant: `serverboot.go` seeds one org per `internal/serverboot/orgid.go` and threads its ID into `core.New(st, tenantID, layers)` (`internal/serverboot/serverboot.go` and `pkg/registry/server/server.go`), and the `core.Registry` consults that one tenant for every request's layer list and visibility (`pkg/registry/core/core.go`, `ListLayerConfigs` and `ListManifests`). Per-request tenant selection from a verified `org_id` is unimplemented; rebinding that `core.Registry` tenant per request is the substrate this proposal depends on. The webhook-receiver and quota tenant is the separate `server.go` `tenant` field (`s.quota.AllowSearch(s.tenant)` at `server.go:699`, `s.webhooks.Deliver(..., s.tenant, ...)` at `events.go:203`), which stays at its `"default"` value in production because the `WithTenant` option that would set it is defined (`server.go:214-216`) but invoked only in tests, never in the production boot path: `bootstrapOptions` and the subsequent `bootOpts` appends in `internal/serverboot/serverboot.go` never add it, so `server.New` falls back to the `tenant: "default"` default (`server.go:227`). It is not a per-request wiring point today either. The gateway-delegated providers in multi-tenant mode depend on the `core.Registry` tenant becoming per-request; a single-tenant deployment, whether standalone or a standard backend booting the single `default` tenant, does not depend on per-request `org_id` routing, because it resolves every request against its sole tenant. Multi-tenant mode itself rests on the multi-tenant-mode startup signal stated in §6.3.3 (more than one provisioned tenant, or an explicit operator opt-in), which the current binary does not yet provide; that signal and the per-request `core.Registry` rebinding land together, so the multi-tenant bind check (§13.10) cannot be reached until both arrive. The single-tenant case still depends on the separate fail-closed-registration prerequisite (§6.3.3): both values are unregistered on the current binary, so a server that selects either one is not yet buildable without serving an anonymous-public registry, regardless of tenancy.

## Spec amendment: §2.2 and §9.1 IdentityProvider built-in enumeration

This subsection drafts the spec changes for `spec/02-architecture.md` and `spec/09-extensibility.md`, which enumerate the `IdentityProvider` built-in value set and are left incomplete by the new values. The §2.2 pluggable-interface bullet and the §9.1 SPI table row both list only `oauth-device-code` and `injected-session-token`. spec/ is read-only, so the proposal carries these amendments, mirroring the §13.12 amendment that revises spec/13-deployment.md:453.

Amend the §2.2 `IdentityProvider` bullet (spec/02-architecture.md:99) to:

> **IdentityProvider**: supplies the OAuth-attested identity attached to every registry call. Client-side built-ins: `oauth-device-code` and `injected-session-token`. Registry-process built-ins: `oidc-jwt` and `trusted-headers` (§6.3.3). Additional implementations register through the interface.

Distinguishing client-side acquisition from registry-side verification matches the registry-process-only framing throughout this proposal and avoids restating a now-incomplete two-value list.

Amend the §9.1 `IdentityProvider` SPI-table row (spec/09-extensibility.md:20) so the Default and Purpose cells cover the registry-process values:

- Default: `` `oauth-device-code` (client-side); `injected-session-token` and the gateway-delegated `oidc-jwt` / `trusted-headers` registry-side (§6.3.3) ``
- Purpose: Attaches OAuth-attested identity to every registry call. Client-side providers acquire the token at the consumer; registry-side providers verify or resolve identity from the request at the registry.

The Purpose cell drops "from the MCP server", because the gateway-delegated values verify or resolve identity at the registry with no MCP server in the request path.

## 13.12 (additions)

- **`PODIUM_IDENTITY_PROVIDER`** gains the registry-process values `oidc-jwt` and `trusted-headers`, alongside `oauth-device-code` and `injected-session-token` (§6.3). These two are read by the registry process only; the MCP server's value-set is unchanged and continues to admit only `oauth-device-code` and `injected-session-token`. Available on a standalone or a standard backend, and mutually exclusive with public mode.
- **`identity_provider.issuer`** (`oidc-jwt`), with `PODIUM_OAUTH_ISSUER` as the environment override. OIDC issuer URL of the IdP that signs the forwarded token. Must use the `https` scheme; the registry rejects a non-`https` value at startup with `config.invalid_issuer_scheme` (§13.12 spec amendment). The registry validates `iss` against this value and derives the JWKS from the issuer's discovery document at `${issuer}/.well-known/openid-configuration`. This is a new key in the `identity_provider:` block, distinct from the top-level domain-discovery `discovery:` block (§4.5.5), which configures domain-tree rendering and is unrelated to identity. `oidc-jwt` reuses `PODIUM_OAUTH_AUDIENCE` (§6.3) for `aud` and adds no audience setting of its own. `PODIUM_OAUTH_AUDIENCE` is required under `oidc-jwt`: when it is unset, the registry rejects startup with `config.oidc_jwt_audience_unset`, so the required `aud` claim is always verified.
- **`PODIUM_OAUTH_TOKEN_HEADER`** (`oidc-jwt`), the environment override for the config-file key `identity_provider.token_header`. Header carrying the forwarded JWT; defaults to `Authorization`. The registry parses the named header's value as `Bearer <token>` regardless of the header name (the `Bearer ` prefix matched case-insensitively, and surrounding whitespace trimmed from `<token>`), so a custom header value must carry the same `Bearer ` prefix as the default; a value that lacks the prefix carries no token and the request is anonymous (§6.3.3). Read only when the selected provider is `oidc-jwt`. Set to a gateway-specific header (for example `X-Podium-Forwarded-JWT`) when the proxy forwards the token elsewhere, and configure the gateway to write `Bearer <token>` into that header.
- **`PODIUM_OAUTH_JWKS_CACHE_TTL_SECONDS`** (`oidc-jwt`), the environment override for the config-file key `identity_provider.jwks_cache_ttl_seconds`. Maximum age in seconds of the cached issuer JWKS before the registry refreshes it from the issuer's discovery document; defaults to 300. Read only when the selected provider is `oidc-jwt`. A `kid` absent from the cached set forces a refresh before this interval elapses.
- **`PODIUM_TRUSTED_PROXY_SECRET`** (`trusted-headers`). Shared secret the gateway sends in `X-Podium-Proxy-Secret`. When set, identity headers are honored only on a request whose secret matches; when unset, the check is skipped and identity headers are honored on every request.

The §6.3.1 `IdpGroupMapping` adapter applies to `oidc-jwt`, which maps the token's group claim. SCIM-based group resolution (§6.3.1) also applies to `oidc-jwt` on a standard (§13.1) backend, but §13.10 scopes SCIM out of standalone, so on a standalone server `oidc-jwt` resolves groups through `IdpGroupMapping` alone. `trusted-headers` reads groups from the request header and consults neither. An operator who has provisioned SCIM should note that `trusted-headers` ignores it: there is no token for the §6.3.1 `IdpGroupMapping` adapter to read, and the gateway is the source of truth, so the registry takes groups from `X-Podium-User-Groups` alone. Provision groups at the gateway for a `trusted-headers` deployment.

The `trusted-headers` identity headers (`X-Podium-User-Sub`, `X-Podium-User-Email`, `X-Podium-User-Groups`, and `X-Podium-User-Org`) are a fixed part of the §6.3.3 contract and carry no environment override. `X-Podium-User-Org` selects the §4.7.1 tenant on a multi-tenant registry, which a standard (§13.1) backend provides. On a standalone (§13.10) backend the registry stays single-tenant, because §13.10 scopes multi-tenancy out of standalone and enabling `oidc-jwt` or `trusted-headers` does not change that; there `X-Podium-User-Org` is not consulted for tenant selection and every request resolves against the sole tenant.

The `auth.untrusted_runtime` §6.9 row and §6.10 envelope are unchanged.

Replace the `injected-session-token` expiry row of the §6.9 failure-modes table so it names the code, and add the rows for the new untrusted-token and tenant-unknown codes. The replaced row reads `Surface "token expired"; the host's runtime is responsible for refresh.` today.

## Spec amendment: §13.10 standalone identity provider, bind constraint, and §13.11 capability and migration passages

This subsection drafts the spec changes for `spec/13-deployment.md` that the standalone availability of `oidc-jwt` and `trusted-headers` requires. Several passages predate this proposal and describe standalone identity as absent or identity filtering as remote-only: the §13.10 topology table describes standalone identity as absent, the §13.11.3 capability list names OIDC identity-based visibility filtering as requiring specifically a remote server, and the §13.11.6 migration triggers name it as a reason to migrate to a remote server. The amendments narrow each passage to the zero-config standalone default and add the gateway-delegated exception; they do not change the default itself, which stays unauthenticated and loopback-bound. A further amendment adds a provider-scoped §13.10 subsection, "Bind restriction under `trusted-headers`", for the `trusted-headers` bind check, which is a new startup requirement that no §13.10 passage covers today. The check is scoped to the `trusted-headers` provider rather than placed in the public-mode "Safety constraints" list, because that list makes `trusted-headers` a startup failure in public mode.

### §13.10 topology table

The current "Identity provider" row of the §13.10 "What changes from the standard topology" table reads:

| Concern           | Standard | Standalone                                      |
| ----------------- | -------- | ----------------------------------------------- |
| Identity provider | OIDC IdP | None. No auth; `127.0.0.1`-only HTTP by default |

Replace the standalone cell so it states the zero-config default and the gateway-delegated option:

| Concern           | Standard | Standalone                                      |
| ----------------- | -------- | ----------------------------------------------- |
| Identity provider | OIDC IdP | None by default. No auth; `127.0.0.1`-only HTTP. A standalone server fronted by a gateway can set `PODIUM_IDENTITY_PROVIDER=oidc-jwt` or `trusted-headers` to filter by gateway-delegated identity (§6.3.3); `trusted-headers` on a non-loopback bind fails to start with `config.trusted_headers_public_bind` unless a proxy secret or `--allow-public-bind` is set (§13.10 "Bind restriction under `trusted-headers`"). |

The default for a standalone server without an identity provider is unchanged: no authentication, `127.0.0.1`-only HTTP, and public visibility for every layer (§13.10). Setting `oidc-jwt` or `trusted-headers` is an explicit opt-in that the cell now records.

The gateway-delegated providers add identity-based visibility filtering to standalone. They do not add multi-tenancy, which §13.10 keeps out of scope for standalone. A standalone server seeds a single tenant at boot and stays single-tenant whether or not a gateway-delegated provider is enabled. The registry resolves every request against the sole tenant and does not consult `org_id` or `X-Podium-User-Org` for tenant selection: the seeded org is synthetic, so an external identity's org value would not name it, and requiring a match would reject or strand otherwise-valid callers. Multi-tenant routing by `org_id` claim or `X-Podium-User-Org` header requires a standard (§13.1) backend and the per-request routing the §6.3.1 amendment above defines, which the registry core does not yet implement.

**Residual cross-org admission.** On a standalone backend the registry maps every verified caller to its sole tenant without consulting `org_id`, so under `oidc-jwt` the configured `PODIUM_OAUTH_AUDIENCE` is the only organization boundary. Any subject for whom the fronting IdP issues a token bearing this audience is treated as an authenticated member of the standalone org. Such a caller can read every `organization: true` layer (§4.6) and any `groups:`/`users:` layer whose listed values match the token's claims, even if the subject belongs to a different organization at the IdP. Layers marked `private` with an explicit `users:` list, or `groups:`, are reachable only by callers whose claims match those values. When the fronting IdP issues the same audience to more than one organization (for example Auth0 Organizations, an Entra multi-tenant application, or an Okta resource audience shared across orgs), scope `PODIUM_OAUTH_AUDIENCE` to this server alone and configure the IdP or gateway to issue that audience only to the intended population. Under `trusted-headers` the gateway is the sole admission boundary, because the registry honors the injected identity without verifying it, so the gateway must restrict which callers it forwards. Multi-tenant routing by verified `org_id` requires a standard backend (§13.1).

**Implementation prerequisite (fail-closed registration).** Standalone availability of these values depends on the same fail-closed-registration prerequisite the §6.3.3 material states. `oidc-jwt` and `trusted-headers` do not exist in the current binary: `identity.Default` registers only `oauth-device-code` and `injected-session-token` (`pkg/identity/registry.go`), so `selectIdentityProvider` resolves no provider for either value, `identityVisibilityGuard` treats them as the externally-fronted exempt case (`!providerSelected`) and does not refuse startup, and the registry falls back to the anonymous-public resolver (`server.New`). On a standalone build that predates the implementation, selecting either value starts a registry that resolves every caller as anonymous-public and applies no per-layer visibility, with no verifier, no error, and no startup warning. The implementation must make both values fail closed before release, so that selecting one on a build without the verifier or resolver refuses to start (`config.identity_provider_unverified`) rather than serving an open registry.

Enabling `oidc-jwt` or `trusted-headers` changes the resolved default layer visibility. When `PODIUM_DEFAULT_LAYER_VISIBILITY` is unset, a standalone server with no identity provider defaults endpoint-registered and `PODIUM_LAYER_PATH` bootstrap layers to public (§13.10); selecting an identity provider resolves the unset default to private (§13.12) instead, so that once the verifier or resolver lands and the registry begins filtering by caller identity, admin-defined layers are not already public to every caller by accident. A standalone server re-ingests its `PODIUM_LAYER_PATH` bootstrap layers on each boot and stamps them with the resolved default, so a layer that carried no explicit `visibility:` setting and was public under the zero-config default becomes private on the first boot after a gateway-delegated provider is enabled, reachable only through an explicit grant. An operator migrating an existing public standalone catalog who wants those layers to stay public must set `PODIUM_DEFAULT_LAYER_VISIBILITY=public`, which the registry applies verbatim regardless of the identity provider, or mark each layer's `visibility:` explicitly.

### §13.12 Layers visibility default

The §13.10 standalone public default is stated unconditionally in three places that predate the gateway-delegated providers, and the resolved-private behavior above (`serverboot.go` guard) narrows each one. Just as the §13.12 amendment above revises the §13.12 "Identity provider" deferral so it no longer claims every identity variable lives in §6.3, this amendment revises the three "public default" passages so each states the identity-provider exception. The narrowing is the same in all three: the unset default resolves to `private` only when an identity provider is set and `PODIUM_DEFAULT_LAYER_VISIBILITY` is unset; an explicit `=public` is applied verbatim regardless of the identity provider.

Revise the §13.12 Layers `PODIUM_DEFAULT_LAYER_VISIBILITY` bootstrap sentence (spec/13-deployment.md, the "Bootstrap layers carry `visibility: public`" sentence) to:

> Without an identity provider, bootstrap layers carry `visibility: public` per the §13.10 standalone default. Once an identity provider (§6.3) is enabled and `PODIUM_DEFAULT_LAYER_VISIBILITY` is unset, the resolved default is `private`, and a bootstrap layer that carried no explicit `visibility:` is re-stamped private on the next boot (§13.10).

Revise the `PODIUM_DEFAULT_LAYER_VISIBILITY` default-cell parenthetical in the same §13.12 Layers table to:

> `private` (§13.10 specifies `public` for a standalone deployment without an identity provider; the unset default resolves to `private` once an identity provider is enabled)

Revise the §13.10 "Layer visibility" default sentence (spec/13-deployment.md, "New layers registered via `podium layer register` default to `visibility: public`") to:

> New layers registered via `podium layer register` default to `visibility: public` on a standalone server without an identity provider; once an identity provider (§6.3) is enabled the unset default resolves to `private` (§13.10). Override with `PODIUM_DEFAULT_LAYER_VISIBILITY`, which the registry applies verbatim regardless of the identity provider.

Each rewrite keeps "when `PODIUM_DEFAULT_LAYER_VISIBILITY` is unset" explicit, so an operator-set `=public` is never narrowed: an identity provider changes only the unset default, and a configured `=public` survives unchanged.

### §13.10 Bind restriction under `trusted-headers`

This subsection drafts a new §13.10 amendment for `spec/13-deployment.md`. It adds a provider-scoped subsection titled "Bind restriction under `trusted-headers`" next to the standalone gateway-delegated identity material above (the topology-table cell and the §6.3.3 cross-reference). The bind rule belongs with the gateway-delegated identity providers rather than with the public-mode "Safety constraints" list. That list is scoped under public mode, and its first bullet makes `PODIUM_IDENTITY_PROVIDER=trusted-headers` a startup failure when public mode is also set, so a `trusted-headers` bind rule placed there would contradict the list it joins. The public-mode "Safety constraints" list is unchanged, including the "Loopback bind by default" bullet, which continues to govern public mode.

`trusted-headers` reads identity from headers it cannot verify, so it carries a bind rule with its own startup checks and error codes. Add the following as a §13.10 subsection adjacent to the gateway-delegated identity material:

> **Bind restriction under `trusted-headers`.** The registry determines at startup whether it is running in multi-tenant mode (more than one provisioned tenant, or an explicit operator opt-in), and the standard backend (§13.1) is not by itself multi-tenant: a standard deployment booting the single `default` tenant is single-tenant and falls under the single-tenant rule, because its `X-Podium-User-Org` resolves to the one tenant and the cross-tenant forging the multi-tenant rule guards against does not exist. When `PODIUM_IDENTITY_PROVIDER=trusted-headers` on a registry running single-tenant, including a standalone backend and a standard backend with only the `default` tenant, the registry binds a loopback address (`127.0.0.0/8`, `::1`) without further configuration, because the registry resolves every request to its sole tenant and does not consult `X-Podium-User-Org`, so a co-resident process cannot cross tenants by forging the header. A non-loopback bind fails to start with `config.trusted_headers_public_bind` unless the operator sets `PODIUM_TRUSTED_PROXY_SECRET` or passes `--allow-public-bind`. On a registry running in multi-tenant mode, where `X-Podium-User-Org` selects among tenants, the loopback exemption does not apply: a loopback bind in a shared-network-namespace pod is reachable by every co-resident container on `127.0.0.1`, so co-residency does not authenticate the gateway, and the registry requires `PODIUM_TRUSTED_PROXY_SECRET` regardless of bind address, including loopback. When the secret is unset in multi-tenant mode, the registry refuses to start with `config.trusted_headers_multitenant_no_secret`. The proxy secret gates header trust at the request level. `--allow-public-bind` records the operator's assumption that an upstream control the registry cannot verify (mutual TLS at the load balancer or mesh, a firewall, or a network policy) keeps the registry reachable only through the gateway. The check runs once at startup and complements the per-request `X-Podium-Proxy-Secret` comparison (§6.3.3). The registry serves HTTP and cannot require or verify a client certificate, because TLS terminates upstream, so the proxy secret is its only request-level control over header trust.

The multi-tenant-mode detection signal is a second startup-time prerequisite alongside the per-request `org_id`-routing substrate the §6.3.1 amendment lists, so the multi-tenant bind check cannot be reached until both the multi-tenant-provisioning path (more than the single `default` tenant, or an explicit opt-in) and the per-request `core.Registry` tenant rebinding land. On the current binary every `trusted-headers` deployment is single-tenant, so only the single-tenant bind rule applies.

This rule is a constraint of the `trusted-headers` identity provider rather than of public mode. It applies only when `trusted-headers` is the selected provider, and the `oidc-jwt` provider carries no bind restriction, because it verifies every token against the issuer's signing key regardless of the network path. The public-mode "Loopback bind by default" bullet and this `trusted-headers` rule never apply to the same registry process, because public mode and an identity provider are mutually exclusive (§13.10).

Because `trusted-headers` is mutually exclusive with public mode (§13.10), the `--allow-public-bind` flag and `PODIUM_ALLOW_PUBLIC_BIND` env var, registered today for public mode (the flag's help text reads "allow public mode to bind a non-loopback address" at `cmd/podium/serve.go:24`), broaden to also override the `trusted-headers` bind check. Update the flag's registered help text so `podium serve --help` is accurate for a `trusted-headers` operator: "allow public mode or trusted-headers to bind a non-loopback address (overrides PODIUM_ALLOW_PUBLIC_BIND)". The same flag and env var keep their public-mode meaning, and the broadened text covers both selections.

This proposal defines `config.trusted_headers_public_bind` and `config.trusted_headers_multitenant_no_secret` as new startup error codes, emitted by the `trusted-headers` startup check drafted above. They join the other startup config codes the registry already emits or this proposal adds: `config.public_mode_with_idp`, defined in §13.10 (the "Mutually exclusive with an identity provider" public-mode Safety-constraints bullet), and `config.invalid_issuer_scheme` and `config.oidc_jwt_audience_unset`, both defined by the §13.12 amendments above (the issuer-scheme requirement and the required-audience requirement for `oidc-jwt`). The registry emits `config.trusted_headers_public_bind` at startup when a non-loopback bind under `trusted-headers` carries neither a proxy secret nor `--allow-public-bind`, names the non-loopback bind address it refused (for example `0.0.0.0:8080`), and directs the operator to set `PODIUM_TRUSTED_PROXY_SECRET` (so the registry trusts the `X-Podium-User-*` headers only on a request carrying the matching `X-Podium-Proxy-Secret`) or to pass `--allow-public-bind`. The registry emits `config.trusted_headers_multitenant_no_secret` when `trusted-headers` is selected on a registry running in multi-tenant mode (more than one provisioned tenant, or an explicit operator opt-in per the §6.3.3 startup signal) with `PODIUM_TRUSTED_PROXY_SECRET` unset, regardless of bind address including loopback, and directs the operator to set the proxy secret because `X-Podium-User-Org` selects among tenants and co-residency does not authenticate the gateway. The registry logs the refusal and does not start. Both codes are boot-time refusals surfaced to stderr; neither is a §6.10 HTTP envelope and neither requires an `errorCodeRegistry` entry, so neither carries a `retryable` or `suggested_action` field, consistent with how §13.10 renders `config.public_mode_with_idp` and `config.layer_path_ambiguous` and how this proposal renders `config.invalid_issuer_scheme`.

### §13.10 web UI under the gateway-delegated providers

The §13.10 web-UI authentication sentence splits two cases today: a standalone deployment without an identity provider serves an open UI on the loopback bind, and a standard deployment uses the same OIDC device-code flow as the CLI with the verification-URL handoff handled in-browser. The standalone availability of `oidc-jwt` and `trusted-headers` adds a third case the sentence does not cover, a standalone deployment whose identity is delegated to a fronting gateway. Amend the sentence so it also covers the gateway-delegated providers:

> Under `PODIUM_IDENTITY_PROVIDER=oidc-jwt` or `trusted-headers`, the web UI is served by the same registry process behind the same gateway and carries no device-code flow of its own. The gateway authenticates the request and the registry resolves the §4.6 caller identity from the forwarded token (`oidc-jwt`) or the injected `X-Podium-User-*` headers (`trusted-headers`), exactly as for any other HTTP API request (§6.3.3). The UI is a thin client over that API, so it inherits the request's resolved identity and performs no token verification itself.

The existing two cases are unchanged; this adds the missing third.

**Bind-guard interaction.** The §13.10 web-UI non-loopback bind guard requires both `--web-ui-allow-public-bind` and a configured identity provider. Its "an identity provider is configured" condition (the `hasIdP` check at `pkg/registry/server/config_validate.go:78`) is satisfied by `oidc-jwt` and `trusted-headers`, because each sets `PODIUM_IDENTITY_PROVIDER` to a non-empty value other than `none`. This is intended: the gateway fronting the registry authenticates the UI's traffic on the same path as every other request, so the UI is no longer unauthenticated once a gateway-delegated provider is selected.

The registry's web-UI handler performs no UI-level authentication of its own, so for `trusted-headers` specifically the operator must still back the gateway-only-reachability assumption. A non-loopback web-UI bind under `trusted-headers` is also subject to the §13.10 "Bind restriction under `trusted-headers`" check, so it requires `PODIUM_TRUSTED_PROXY_SECRET` or `--allow-public-bind` in addition to `--web-ui-allow-public-bind`. Without one of those, a gateway-bypass request, which §6.3.3 states carries no trusted identity, would otherwise reach a publicly-bound `/ui/` as an unauthenticated, public-visibility session. The two checks compose: `--web-ui-allow-public-bind` covers the UI mount, and the `trusted-headers` bind restriction covers header trust on the same non-loopback bind. Under `oidc-jwt`, which verifies every token regardless of the network path, the web-UI guard's identity-provider condition is the only added requirement, because `oidc-jwt` carries no bind restriction.

### §13.11.3 capability list

The §13.11.3 "What's Available" section lists OIDC identity-based visibility filtering under "Features that require **specifically a remote server** (not just any server)", alongside centralized audit and multi-tenancy. The standalone availability of `oidc-jwt` and `trusted-headers` makes that entry inaccurate, in the same way it does the §13.11.6 trigger below. The current list reads:

> Features that require **specifically a remote server** (not just any server):
>
> - Centralized audit independent of clones.
> - OIDC identity-based visibility filtering.
> - Multi-tenancy, SCIM, transparency-log anchoring.

Replace the identity-filtering entry so it states the gateway exception, and leave centralized audit and multi-tenancy as the remaining remote-only entries:

> Features that require **specifically a remote server** (not just any server):
>
> - Centralized audit independent of clones.
> - Multi-tenancy, SCIM, and transparency-log anchoring.
>
> Identity-based visibility filtering requires a server. It does not require a remote server specifically: a standalone server fronted by a gateway provides it through `oidc-jwt` or `trusted-headers` (§6.3.3), and a remote standard deployment runs its own OIDC IdP. Filesystem source has no server and no identity, so its visibility evaluator short-circuits to `true` (the "not available in filesystem source" list above).

### §13.11.6 migration triggers

The current §13.11.6 migration triggers name OIDC identity-based visibility filtering as a remote-server reason, and read:

> **Migrate to any server (local `podium serve --standalone` or remote standard deployment):**
>
> - Progressive disclosure required (agents call MCP meta-tools at runtime to load capabilities incrementally instead of materializing everything ahead of time).
>
> **Migrate specifically to a remote server:**
>
> - Centralized audit independent of clones.
> - OIDC identity-based visibility filtering.

Move identity-based filtering to the "any server" cluster, qualified by the gateway requirement, and leave centralized audit as the remaining remote-only trigger:

> **Migrate to any server (local `podium serve --standalone` or remote standard deployment):**
>
> - Progressive disclosure required (agents call MCP meta-tools at runtime to load capabilities incrementally instead of materializing everything ahead of time).
> - Identity-based visibility filtering. A standalone server fronted by a gateway sets `PODIUM_IDENTITY_PROVIDER=oidc-jwt` or `trusted-headers` and filters by the gateway-delegated identity (§6.3.3). A remote standard deployment runs its own OIDC IdP instead.
>
> **Migrate specifically to a remote server:**
>
> - Centralized audit independent of clones.

A standalone server without a gateway still has no identity provider, so identity-based filtering on a bare standalone server remains a reason to add a gateway or to migrate. The trigger moves because the filtering no longer requires a remote standard deployment once a gateway fronts the standalone server.

## Example configuration

Configuration lives in `registry.yaml` (§13.12), and an environment variable overrides the file value of the same setting. A secret stays in the environment rather than the file.

### oidc-jwt

```yaml
# registry.yaml  (standalone or standard server, fronted by a gateway)
identity_provider:
  type: oidc-jwt
  issuer: https://acme.okta.com/oauth2/default   # must be https; JWKS derived from ${issuer}/.well-known/openid-configuration
  audience: https://podium.acme.com
  token_header: Authorization   # default; value parsed as "Bearer <token>" for any header name
  jwks_cache_ttl_seconds: 300   # default; max age of the cached issuer JWKS
  # Group mapping is configured as in §6.3.1. SCIM applies on a standard backend; a standalone server resolves groups through IdpGroupMapping alone.
```

Gateway responsibility: forward the caller's IdP-signed JWT to the registry in the configured header as `Bearer <token>`, whether the header is the default `Authorization` or a custom one such as `X-Podium-Forwarded-JWT`. Header stripping is unnecessary, because a forged token fails verification.

### trusted-headers

```yaml
# registry.yaml  (standalone or standard server, fronted by a gateway)
identity_provider:
  type: trusted-headers
  # The proxy secret is read from PODIUM_TRUSTED_PROXY_SECRET, not stored here.
  # On a registry running single-tenant (a standalone backend, or a standard
  # backend with only the default tenant), a non-loopback bind requires a
  # proxy secret or --allow-public-bind; otherwise startup fails with
  # config.trusted_headers_public_bind.
  # On a registry running in multi-tenant mode, X-Podium-User-Org selects among
  # tenants, so the proxy secret is required regardless of bind address,
  # including loopback; otherwise startup fails with
  # config.trusted_headers_multitenant_no_secret.
```

Gateway responsibility: authenticate the caller, remove any client-supplied `X-Podium-User-*` headers, set the identity headers from the authenticated session (including `X-Podium-User-Org` on a multi-tenant registry), and, when a secret is configured, attach `X-Podium-Proxy-Secret`. The registry must be reachable only through the gateway. The proxy secret enforces that at the request level. On a registry running single-tenant (a standalone backend, or a standard backend with only the `default` tenant), a non-loopback bind without a proxy secret requires `--allow-public-bind`, which records the operator's assumption that an upstream control keeps the registry reachable only through the gateway. On a registry running in multi-tenant mode, where `X-Podium-User-Org` selects among tenants, `PODIUM_TRUSTED_PROXY_SECRET` is required regardless of bind address, including loopback; `--allow-public-bind` does not substitute for it, and startup fails with `config.trusted_headers_multitenant_no_secret` when the secret is unset.

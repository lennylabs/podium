# Proposal 0002: Runtime Tenant Provisioning

- Issue: (to be filed)
- Status: Implemented
- Date: 2026-06-08

## Summary

A multi-tenant registry (§4.7.1, §6.3.1) resolves each request to a provisioned tenant and rejects a request whose organization names no provisioned tenant with `auth.tenant_unknown` (§6.10). The registry has no runtime mechanism to provision a tenant. Tenants are created only at process boot: `provisionTenants` reads the `PODIUM_TENANTS` environment variable and calls `store.CreateTenant` for each name, and `bootstrapDefaultTenant` creates the single `default` org (`internal/serverboot/orgid.go:23`, `:84`; `internal/serverboot/serverboot.go:701-705`). Adding an organization therefore requires editing `PODIUM_TENANTS` and restarting the registry. There is no admin API route and no CLI command that creates, lists, updates, or deactivates a tenant: the registered routes carry `/v1/admin/grants`, `/v1/admin/reembed`, `/v1/admin/show-effective`, and `/v1/admin/erase`, but no tenant route (`pkg/registry/server/server.go`), and the admin CLI dispatches `erase`, `retention`, `reembed`, `grant`, `revoke`, `show-effective`, `runtime`, and `migrate-to-standard`, but no `tenant` verb (`cmd/podium/admin.go`).

This proposal adds runtime tenant provisioning: a set of operator-authorized admin API endpoints under `/v1/admin/tenants` and a `podium admin tenant` CLI group, backed by the existing `tenants` table. The store already persists tenants and, on the Postgres backend, `CreateTenant` provisions the org's schema and row-level-security policy before inserting the row (`pkg/store/postgres.go:456-478`), so the data-plane substrate exists. The gap is the store read and write surface (the interface exposes only `CreateTenant` and `GetTenant`, `pkg/store/store.go:340-341`), an operator-level authorization distinct from the per-tenant admin check, the API endpoints, and the CLI. This proposal also removes `PODIUM_TENANTS` and the boot-time `provisionTenants` path: the runtime API and CLI become the only way to provision a tenant, and the registry creates no tenant from configuration at boot beyond the single `default` org (`bootstrapDefaultTenant`). The sections below are drafted for `spec/04-artifact-model.md`, `spec/07-external-integration.md`, `spec/06-mcp-server.md`, and `spec/13-deployment.md`; `spec/` is read-only, so this proposal carries the amendments to be applied on acceptance, as proposal 0001 did.

## Current state and the gap

The tenant boundary is the org (§4.7.1). Each tenant carries an ID (a UUID), a human-readable name, a per-tenant `Quota` (storage bytes, search QPS, materialize rate, audit volume per day, and the §7.3.1 user-layer cap), and the §3.5 `ExposeScopePreview` gate (`pkg/store/store.go:53-76`). Org IDs are name-derived UUIDv5 values so they stay stable across restarts (`internal/serverboot/orgid.go:72`).

What exists:

- **Persistence.** The `tenants` table on SQLite (`pkg/store/sqlite.go:69`) and `public.tenants` on Postgres (`pkg/store/postgres.go:159`). `CreateTenant` is idempotent (`INSERT OR IGNORE` / `ON CONFLICT (id) DO NOTHING`), and on Postgres it provisions the org's schema and writes under the per-request `podium.org_id` so the row satisfies the row-level-security `WITH CHECK` (`pkg/store/postgres.go:456-478`).
- **Per-request routing.** Proposal 0001 (§6.3.1) added per-request `org_id`-to-tenant selection, with `tenantResolver` mapping an organization value to a provisioned tenant (`internal/serverboot/orgid.go:43`).
- **Per-tenant admin.** `requireAdmin` authorizes a request against the admin grants of the request's tenant through `core.AdminAuthorize` (`pkg/registry/server/admin.go:113`). `PODIUM_BOOTSTRAP_ADMINS` seeds tenant-admin grants for the bootstrapped tenant at boot (`internal/serverboot/serverboot.go:347`).

What is missing:

- **Store read and write surface.** The `store.Store` interface exposes `CreateTenant` and `GetTenant` only. There is no `ListTenants`, no `UpdateTenant`, and no deactivation method (`pkg/store/store.go:340-341`).
- **Runtime provisioning path.** `CreateTenant` is called only from `provisionTenants` and `bootstrapDefaultTenant` at boot, from `NewFromFilesystem` for the standalone filesystem bootstrap, and from the `migrate-to-standard` store copy (`cmd/podium/admin_migrate.go:354`). None is an operator-facing "add a tenant to a live registry" path.
- **Operator-level authorization.** `requireAdmin` is per-tenant: it checks an admin grant scoped to the request's org. Provisioning a new org, and listing every org, crosses tenant boundaries, so it cannot be authorized by a grant inside a single tenant. The registry has no instance-level operator role today.
- **Per-tenant configuration at runtime.** `provisionTenants` creates a tenant with its name and the process-wide `ExposeScopePreview` value, and leaves the `Quota` at its zero value. There is no path to set a per-tenant quota, and no path to change a tenant's configuration after boot.

## Proposed solution

### Operator authorization

Tenant provisioning is an instance-level operation. This proposal introduces an operator authorization distinct from the per-tenant `requireAdmin`. An operator is an identity that the deployment designates as able to manage tenants across the instance, established the same way bootstrap admins are: a `PODIUM_OPERATOR_ADMINS` environment variable (a comma-separated list of identities, mirroring `PODIUM_BOOTSTRAP_ADMINS`) seeds instance-operator grants at boot. The grant is stored in a dedicated cross-org table (`public.operator_grants` on Postgres, a flat `operator_grants` table on SQLite/Memory), read and written through a privileged path that does not transit the per-org schema router (`p.org`/`orgSchemaIdent`), mirroring how `public.vector_pending` is a cross-org table outside any single org's schema. This is required because the per-org admin-grants path validates the org id against `orgIDPattern` (`pkg/store/postgres.go:47`) and provisions a per-org schema via `ensureOrg`; an instance-level operator grant is not scoped to any org schema. The operator-grant key is an identity, with no org id, so it neither collides with a name-derived org nor manufactures a phantom org schema. A new `core.OperatorAuthorize(ctx, identity)` checks for that grant, and a new `Server.requireOperator(r)` gates every `/v1/admin/tenants` endpoint. A request that lacks operator authorization is rejected with `auth.forbidden` (§6.10), the code `requireAdmin` already returns.

Because `withTenantRouting` (`pkg/registry/server/server.go:401`) runs before the handler mux and rejects a caller whose `org_id` resolves to no provisioned tenant with `auth.tenant_unknown`, the `/v1/admin/tenants` routes bypass tenant routing so the operator's request reaches `requireOperator` on a fresh registry. The server skips `withTenantRouting` for these paths (a prefix check on `/v1/admin/tenants` in `withTenantRouting`, mirroring the `pathRequiresIdentity` exemptions for `/healthz`, `/readyz`, and `/scim/`), so the cross-org operator grant is checked at the handler regardless of the caller's organization value.

The operator role is a strict superset of nothing: it authorizes tenant management only. It does not grant per-tenant admin rights inside any org; an operator who needs to administer a specific tenant receives a normal admin grant in that tenant (§4.7.2). The two authorization levels stay separate so that compromising a tenant admin cannot create or read other tenants, and so that an operator credential is not a per-tenant data-plane credential.

### Store interface additions

Add three methods to `store.Store`, implemented on the Memory, SQLite, and Postgres backends:

- `ListTenants(ctx context.Context) ([]Tenant, error)` returns every provisioned tenant. On Postgres, `public.tenants` carries `FORCE ROW LEVEL SECURITY` with a policy of `id = current_setting('podium.org_id', true)` (`pkg/store/postgres.go:202`, `:209`), which subjects even the registry's table-owner connection to the per-org predicate, so a per-`org_id` read returns only that org's row. A cross-org list re-scopes the policy through an operator sentinel that the table owner sets. The proposal extends the FORCE'd policy on `public.tenants` to admit a sentinel value, so the policy reads `id = current_setting('podium.org_id', true) OR current_setting('podium.org_id', true) = '*operator-list*'`. `ListTenants` runs on the registry's existing table-owner connection: it sets `podium.org_id` to the `*operator-list*` sentinel for the read, so the cross-org list returns every row, and it does so without any role holding `BYPASSRLS`. The sentinel is not a valid org ID (`orgIDPattern` admits only UUIDs, `pkg/store/postgres.go:47`), so no `public.tenants` row carries it and it cannot collide with a provisioned org. The existing per-`org_id` `GetTenant`/`CreateTenant` reads stay on the owner connection and stay confined to a single org by the same policy; only `ListTenants` sets the sentinel. Creating, altering, and forcing a policy require only ownership of `public.tenants`, which the registry's `PODIUM_POSTGRES_DSN` role already holds because `applySchema` runs `rlsStmts` from the binary today (`pkg/store/postgres.go:284`, `rlsStmts` at `:200`). On a freshly provisioned database `applySchema` creates the policy with the sentinel predicate; on an already-provisioned database it updates the existing policy in place through an owner-only `CREATE OR REPLACE POLICY`, so a binary upgrade admits the sentinel without a manual step rather than skipping the change the way an unconditional create-if-absent would. The sentinel setup adds no superuser step. The list works on managed Postgres (RDS, Aurora, Cloud SQL, §13.1), where the customer master role owns the table but is not a true superuser.
- `UpdateTenant(ctx context.Context, t Tenant) error` writes a tenant's mutable configuration (`Quota`, `ExposeScopePreview`, and the `Active` flag) and returns `ErrTenantNotFound` when the ID is unknown. It changes neither the ID nor the name. The name is the org-name alias (§4.7.1) and the org ID is its UUIDv5 derivation (`orgIDForName`), so the name is fixed at create: the resolver routes a request by deriving the org ID from the supplied name, and changing the stored name without re-deriving the ID would leave the tenant reachable only under its creation-time name. `UpdateTenant` ignores any `Name` set on the passed `Tenant`. It writes the whole mutable record, so the PATCH handler constructs the merged `Tenant` by overlaying the supplied top-level fields and, for `quota`, the supplied quota sub-fields onto the current row before calling it; a single-field or single-quota-sub-field PATCH does not reset any omitted field or sub-field. Writing `Active` true through `UpdateTenant` reactivates a deactivated tenant; there is no separate reactivation method.
- `DeactivateTenant(ctx context.Context, id string) error` sets the tenant's `Active` flag false without deleting its data (see Lifecycle below), and returns `ErrTenantNotFound` for an unknown ID.

`CreateTenant` keeps its idempotent contract on all three backends: inserting an already-present ID is a no-op that leaves the stored row, including its `Active` flag, unchanged. `Memory.CreateTenant` changes from an unconditional map assignment to an insert-if-absent (it returns without writing when the ID already exists), matching the SQLite `INSERT OR IGNORE` and Postgres `ON CONFLICT (id) DO NOTHING` semantics, so a `POST` create against a deactivated tenant does not reset `Active` and does not reactivate it.

The `Tenant` struct gains an `Active bool` field, backed by an `active` column on the SQLite (`pkg/store/sqlite.go:69-78`, `active INTEGER NOT NULL DEFAULT 1`) and Postgres (`public.tenants`, `active BOOLEAN NOT NULL DEFAULT TRUE`) schemas. The column default applies both to rows that predate the column (additive migration, `pkg/store/schema_migrate.go`) and to a new insert whose column list omits `active`, so a tenant created through the unchanged SQLite `INSERT OR IGNORE` and Postgres `ON CONFLICT (id) DO NOTHING` statements is active without those statements writing the column. The resolver consults this flag: `tenantResolver` (`internal/serverboot/orgid.go:43`) treats a tenant whose `Active` flag is false as unprovisioned and returns false, so a `GetTenant` that returns a deactivated row no longer resolves to it. This is the mechanism currently missing from `internal/serverboot/orgid.go:49-55`, where the resolver returns true whenever `GetTenant` succeeds.

The `CreateTenant` signature and its idempotent semantics, `GetTenant`, and `ErrTenantNotFound` (`registry.tenant_not_found`) are unchanged. `Memory.CreateTenant` is the only `CreateTenant` implementation to change: it adopts the insert-if-absent behavior described above, and, because the Memory backend has no column-default layer and `Active bool` zero-values to false, it stores the row with `Active` set true when the caller leaves it unset, matching the SQL backends' column default. A created tenant is therefore active on all three backends, so the `default` org and every other newly created tenant resolves; `DeactivateTenant` is the only path that sets `Active` false. `GetTenant` reads the `active` column into `Tenant.Active` on the SQL backends so the resolver sees a tenant's current active state.

### Admin API

Add four endpoints, each gated by `requireOperator` and audited (§8.1). The mutating endpoints (`POST`, `PATCH`, and `DELETE`) are write endpoints under §13.2.1, so a read-only registry rejects them with `registry.read_only` after operator authorization and before any store mutation, mirroring `handleAdminGrants`, which calls `rejectIfReadOnly` after `requireAdmin` (`pkg/registry/server/admin.go:31`); `GET /v1/admin/tenants` is a read and stays available in read-only mode.

- **`POST /v1/admin/tenants`** creates a tenant. The request body carries `name` (required) and an optional `quota` object and `expose_scope_preview` boolean. The registry derives the org ID from the name through the existing `orgIDForName` UUIDv5 derivation (`internal/serverboot/orgid.go:72`), so the org ID is stable across restarts. The endpoint reuses `store.CreateTenant`, which provisions the Postgres schema and RLS policy. Because `CreateTenant` is idempotent, repeating a create call for an already-provisioned name returns `200 OK` with the existing tenant and no change rather than a conflict error, so a repeated apply of the same call is safe; the response indicates whether the tenant was created or already existed. The endpoint returns `201 Created` for a new tenant with `{id, name, quota, expose_scope_preview, active}`; the `200 OK` idempotent-create response carries the same fields, so the operator can see whether the already-provisioned tenant is active. A created tenant is active; deactivation is a separate `DELETE`/`PATCH active:false` operation.
- **`GET /v1/admin/tenants`** lists every provisioned tenant via `ListTenants`, returning `{tenants: [...]}`, where each element is the same tenant object `{id, name, quota, expose_scope_preview, active}`. This is the only read that crosses org boundaries, and it is operator-only.
- **`PATCH /v1/admin/tenants/{id}`** is a partial update of a tenant's quota, scope-preview gate, or active state via `UpdateTenant`; it does not change the name (fixed at create, see Store interface additions). The registry reads the current tenant and overlays only the supplied fields: at the top level `quota`, `expose_scope_preview`, or `active`, and within `quota` only the quota sub-fields present in the body, preserving every omitted sub-field. Quota sub-fields are nullable in the body so an explicit `0` is distinguished from omission (§4.7.8 gives `0` a defined meaning). An `active: true` field reactivates a deactivated tenant. An unknown ID returns `404 registry.tenant_not_found`.
- **`DELETE /v1/admin/tenants/{id}`** deactivates a tenant via `DeactivateTenant` (a soft operation, see Lifecycle). An unknown ID returns `404 registry.tenant_not_found`.

### CLI

Add a `podium admin tenant` dispatch group, mirroring the existing `podium admin runtime` nested group (`cmd/podium/admin_runtime.go`), with the operator's identity supplied through the normal client login (the operator authenticates as any caller does, and the registry checks the operator grant):

- `podium admin tenant create <name> [--storage-bytes N] [--search-qps N] [--materialize-rate N] [--audit-volume-per-day N] [--max-user-layers N] [--expose-scope-preview true|false] --registry <url>`
- `podium admin tenant list [--json] --registry <url>`
- `podium admin tenant update <id> [--storage-bytes N] [...] [--active true|false] --registry <url>`
- `podium admin tenant deactivate <id> --registry <url>`

The `update --active true` flag maps to the PATCH `active` field and reactivates a deactivated tenant; `update --active false` deactivates it, the same soft operation as `deactivate`. `podium admin tenant update` sends only the flags the operator passes, so a flag the operator omits is left out of the PATCH body and the corresponding field is preserved server-side. For example, `update <id> --storage-bytes N` changes only the storage quota and preserves the other quota fields, the scope-preview gate, and the active state. `update` cannot change the tenant name, which is fixed at create.

The flags map one-to-one to the `Quota` fields and the `ExposeScopePreview` gate. `list --json` emits the wire array for scripting, consistent with the `--json` convention on the other read commands (`cmd/podium/config.go`).

### Remove the boot-time `PODIUM_TENANTS` path

This proposal removes the `PODIUM_TENANTS` environment variable and the `provisionTenants` boot path (`internal/serverboot/orgid.go:23`, `internal/serverboot/serverboot.go:701-705`). The registry no longer provisions tenants from an environment list; the runtime API and CLI are the only way to provision a tenant beyond the `default` org, which `bootstrapDefaultTenant` always creates at boot. A multi-tenant deployment seeds its first operator through `PODIUM_OPERATOR_ADMINS`, and that operator creates the remaining tenants at runtime, so the boot environment no longer carries a tenant list. `PODIUM_TENANTS` appears nowhere under `spec/`, so its removal is an implementation change with no §13.12 spec-table row to remove. Removing `provisionTenants` does not affect `tenantResolver` (`internal/serverboot/orgid.go:43`), the §6.3.1 per-request routing path, which stays.

### Lifecycle and deletion

Deactivation is soft. A deactivated tenant stops resolving: `tenantResolver` (`internal/serverboot/orgid.go:43`) treats an inactive tenant as unprovisioned, so a request naming it is rejected with `auth.tenant_unknown` under `oidc-jwt` or sees public visibility only under `trusted-headers`, exactly as for a never-provisioned org. The tenant's data (its schema, artifacts, layers, admin grants, and audit history) persists. This matches the §8.4 soft-delete-with-recovery model for user-defined layers, and it honors the §8 audit-retention guarantees (§8.4), which a hard delete would violate by destroying audit history that retention policy may require the operator to keep; the persisting data also stays in the deployment's single region (§4.7.1.1). Reactivation is `PATCH /v1/admin/tenants/{id}` with an `active: true` field, which clears the inactive marker and restores resolution. A `POST /v1/admin/tenants` create call for the same name does not reactivate the tenant: it reuses the idempotent `CreateTenant`, which leaves the existing row, including its inactive marker, unchanged, so reactivation runs only through the `active: true` update.

Hard purge (destroying the org's schema and all of its data) is out of scope for this proposal. It is a destructive, retention-sensitive operation that warrants its own design alongside the §8.5 right-to-erasure path (`podium admin erase`), and it is not required to close the onboarding gap this proposal addresses. A follow-up may add `podium admin tenant purge --confirm` subject to the retention window.

### Quota defaults

A tenant created without an explicit quota carries the zero-value `Quota`. The create and update endpoints let an operator set an explicit per-tenant quota, and they accept the zero value. The interpretation of a zero quota field is defined where the quota is enforced (§4.7.8), which this proposal amends to state the zero-value semantics (a zero `storage`, `search QPS`, `materialization rate`, or `audit volume` disables that budget; a zero `user-layer cap` selects the deployment default and a negative value disables it). Where the deployment requires concrete per-tenant budgets, the operator sets them through the API or CLI at or after creation.

## Spec amendment: §4.7.1 tenant provisioning lifecycle

`spec/04-artifact-model.md:772` (§4.7.1 Tenancy) defines the org as the tenant boundary and defines org IDs as UUIDs with names as aliases, but specifies no provisioning lifecycle: it does not state how a tenant comes to exist, change, or stop resolving. Add a "Provisioning" subsection to §4.7.1:

> **Provisioning.** A tenant is provisioned before any request resolves to it. On a multi-tenant registry (§6.3.1, started with `PODIUM_MULTI_TENANT`), a request whose organization names no provisioned tenant is rejected with `auth.tenant_unknown` (§6.10) under `oidc-jwt`, or sees public visibility only under `trusted-headers` (§6.3.3). Per-request tenant resolution exists only on a multi-tenant registry; a single-tenant registry (§6.3.1) binds every request to its sole tenant and does not consult the organization value. A single-tenant registry has no additional tenant to resolve: it provisions no tenant from configuration beyond the `default` org, and it rejects every `/v1/admin/tenants` provisioning request with `registry.tenant_management_unavailable` (§7.3.3, §6.10). An operator (§4.7.1 Operator role) provisions a tenant at runtime through the `/v1/admin/tenants` API (§7.3.3) or the `podium admin tenant` CLI (§13), and creates, lists, updates, and deactivates tenants there; provisioning is idempotent, so creating an already-provisioned name is a no-op. The registry provisions no tenant from configuration at boot beyond the single `default` org, which it always creates. A deactivated tenant stops resolving and is treated as unprovisioned, while its data, including its audit history, persists per the §8 retention guarantees (audit events retained per §8.4) and stays in the deployment's single region (§4.7.1.1); reactivating it restores resolution. The operator grant is held in a cross-org operator-grant table outside any per-org schema, so it is not scoped to a tenant; reads and writes of the operator grant use a privileged cross-org path.

Add an "Operator role" paragraph to §4.7.1 that defines the instance-level authorization:

> **Operator role.** Provisioning crosses tenant boundaries, so it is authorized at the instance level rather than by a per-tenant admin grant (§4.7.2). An operator is an identity the deployment designates through `PODIUM_OPERATOR_ADMINS` (§13.12), which seeds an instance-operator grant at boot. The operator role authorizes the `/v1/admin/tenants` endpoints only; it confers no per-tenant admin rights, and a per-tenant admin grant confers no operator rights. Because tenant management is an instance-level operation that crosses tenant boundaries, the `/v1/admin/tenants` endpoints are exempt from the §6.3.1 per-request tenant selection: the registry does not resolve the caller's organization to a tenant for these endpoints and does not reject the request with `auth.tenant_unknown` when the caller's `org_id` names no provisioned tenant. On a multi-tenant registry (§6.3.1, started with `PODIUM_MULTI_TENANT`) the endpoints are gated by operator authorization alone, so the first operator can create the first additional tenant even though only the `default` org has been provisioned at boot; a single-tenant registry rejects every `/v1/admin/tenants` request with `registry.tenant_management_unavailable` regardless of operator authorization (§7.3.3). The accept-versus-reject distinction is keyed on the registry mode (`PODIUM_MULTI_TENANT`), never on the count of provisioned tenants. A request that reaches a tenant-management endpoint without operator authorization is rejected with `auth.forbidden`.

Amend the §4.7.1 "Postgres isolation" sentence (`spec/04-artifact-model.md:778`) to distinguish per-org cross-org tables from the instance-scoped operator-grant table. Replace:

> **Postgres isolation.** Each org has its own schema; cross-org tables (e.g., shared infrastructure metadata) use row-level security with org_id checks. Schema-per-org gives clean drop-org semantics, isolates query patterns, and bounds the blast radius of SQL injection.

with:

> **Postgres isolation.** Each org has its own schema. The tenant registry is a per-org cross-org table under forced row-level security whose policy admits a session scoped to one org and an instance operator-list sentinel (§7.3.3), so a session scoped to one org, including the table owner's, reads and writes only that org's registry row, while the registry's operator-list read sets the sentinel on the table-owner connection to read every org's row. The vector outbox is a per-org cross-org table that carries the same org_id check but does not force row-level security, so a non-owner session scoped to one org reads and writes only its own rows while the trusted registry process, which connects as the table owner, reads and writes every org's rows to drain the outbox in one scan (§4.7). An instance-scoped cross-org table that holds no per-org data, such as the operator-grant table (§4.7.1 Operator role), carries no org_id column and cannot use an org_id check; the registry reads and writes it through a privileged cross-org path rather than through the per-org row-level-security predicate. Schema-per-org gives clean drop-org semantics, isolates query patterns, and bounds the blast radius of SQL injection.

## Spec amendment: §7.3.3 tenant-management endpoints

Add a new subsection §7.3.3 (Tenant Management) under §7.3 (Host Integration) in `spec/07-external-integration.md`, after §7.3.2 (Outbound Webhooks). These are operator-level HTTP endpoints, parallel to the §7.3.1 layer-management endpoints (`GET /v1/layers`, `POST /v1/layers/reingest`, `DELETE /v1/layers`). Operator authorization is distinct from the per-tenant `admin` role defined in §4.7.2; the operator role is established per §4.7.1 (Operator role). The four endpoints are operator-only and audited (§8.1). The mutating endpoints (`POST`, `PATCH`, and `DELETE`) are write endpoints under §13.2.1, so a read-only registry rejects them with `registry.read_only`, consistent with the layer-admin and admin-grant endpoints; `GET /v1/admin/tenants` is a read and stays available in read-only mode. The registry checks this after operator authorization, mirroring the admin-grant handler, which calls `rejectIfReadOnly` after `requireAdmin` (`pkg/registry/server/admin.go:31`).

- `POST /v1/admin/tenants` creates a tenant from `{name, quota?, expose_scope_preview?}`, deriving the org ID from the name, reusing the idempotent `CreateTenant`, and returning `201` with the tenant object `{id, name, quota, expose_scope_preview, active}` (or `200` with the same object when the name already names a provisioned tenant, so a re-create surfaces the existing tenant's `active` state).
- `GET /v1/admin/tenants` returns `{tenants: [...]}`, each element the tenant object `{id, name, quota, expose_scope_preview, active}`, the only cross-org read, operator-only. The tenants registry is under `FORCE ROW LEVEL SECURITY`, so the registry re-scopes the policy through an operator sentinel rather than by setting a per-org `org_id`: its FORCE'd policy admits the sentinel value `*operator-list*`, and the read runs on the registry's existing `PODIUM_POSTGRES_DSN` table-owner connection with `podium.org_id` set to that sentinel, returning every row. Every per-org tenant read keeps `podium.org_id` set to a real org ID and stays confined to its own org. The sentinel policy is created from the binary, because creating and altering a policy and forcing row-level security require only table ownership, which the owner connection holds; the schema setup adds no superuser step and the read holds no `BYPASSRLS` attribute, so the list works on managed Postgres (§13.1).
- `PATCH /v1/admin/tenants/{id}` is a partial update. The registry reads the current tenant and merges the supplied fields. At the top level it applies only `quota`, `expose_scope_preview`, or `active` when present. Within `quota`, the merge is per sub-field: it overwrites only the quota sub-fields present in the body (`storage_bytes`, `search_qps`, `materialize_rate`, `audit_volume_per_day`, `max_user_layers`) and preserves every omitted sub-field at its current value. Because a zero quota value is itself meaningful (§4.7.8), each quota sub-field is nullable in the request body: a present sub-field set to `0` is applied as zero, and an absent sub-field is preserved. Omitting `quota` entirely preserves the whole existing quota; omitting `expose_scope_preview` preserves the existing tri-state gate; omitting `active` leaves the active state unchanged. The registry writes the merged record via `UpdateTenant` and returns the merged tenant object `{id, name, quota, expose_scope_preview, active}`. The endpoint does not change the name, which is fixed at create. An unknown ID returns `404 registry.tenant_not_found`.
- `DELETE /v1/admin/tenants/{id}` deactivates the tenant (soft); an unknown ID returns `404 registry.tenant_not_found`.

The org ID and the org-name alias (§4.7.1) are both fixed at create. The org ID is the UUIDv5 derivation of the name (§6.3.1 resolves a request's org-name alias to its org ID through this derivation), so `PATCH` cannot change `name`: changing the stored name without re-keying the org would leave the tenant resolvable only under its creation-time name and unreachable under the new one. Rename is therefore out of scope; a tenant that needs a different name is provisioned anew under that name.

The tenant-management endpoints are available only on a multi-tenant standard backend (§13.1). A single-tenant backend, whether a standalone backend (§13.10) with its sole synthetic `default` org or a standard backend started without `PODIUM_MULTI_TENANT`, rejects every `/v1/admin/tenants` request with `registry.tenant_management_unavailable` (§6.10), because multi-tenancy is out of scope for standalone (§13.10) and the standalone routing does not consult tenants created through the API.

## Spec amendment: §13 CLI and §13.12 variables

Add the `podium admin tenant` group to the §13 operator CLI: `create`, `list`, `update`, and `deactivate`, with the flags listed above, authorized as the operator role (§4.7.1). Add to the §13.12 variable table:

| Var | Description | Default |
| --- | --- | --- |
| `PODIUM_OPERATOR_ADMINS` | Comma-separated identities granted the instance-operator role at boot. The operator role authorizes the `/v1/admin/tenants` tenant-management endpoints (§7.3.3) and the `podium admin tenant` CLI; it confers no per-tenant admin rights. Distinct from `PODIUM_BOOTSTRAP_ADMINS`, which seeds per-tenant admin grants for the bootstrapped tenant. | (unset) |
| `PODIUM_MULTI_TENANT` | When `true`, the registry runs in multi-tenant mode: it binds to a no-data tenant and resolves each request's tenant from the caller's organization (§6.3.1), rejecting an `oidc-jwt` token whose `org_id` names no provisioned tenant with `auth.tenant_unknown`. When unset, the registry binds every request to the single `default` org and does not consult the organization value, and the §7.3.3 tenant-management endpoints are rejected. | (unset → single-tenant) |

**Multi-tenant operator tenant-list (Postgres).** On a multi-tenant Postgres backend, the cross-org `GET /v1/admin/tenants` read (§7.3.3) runs on the registry's existing `PODIUM_POSTGRES_DSN` table-owner connection, with `podium.org_id` set to the `*operator-list*` sentinel that the FORCE'd policy on `public.tenants` admits. The deployment provisions no role or function for this read: extending the FORCE'd policy to admit the sentinel and forcing row-level security require only ownership of `public.tenants`, which the `PODIUM_POSTGRES_DSN` role holds, so the registry's schema setup applies the sentinel policy from the binary with the rest of the row-level-security setup and the list needs no manual step. The registry's `PODIUM_POSTGRES_DSN` role must be the non-superuser owner of `public.tenants`: a superuser DSN role bypasses row-level security entirely, which voids the per-org confinement of every other tenant read.

This proposal removes the `PODIUM_TENANTS` environment variable and the `provisionTenants` boot path (see "Remove the boot-time `PODIUM_TENANTS` path"). `PODIUM_TENANTS` appears nowhere in the current spec, so its removal touches no §13.12 row. The §13.12 additions are the `PODIUM_OPERATOR_ADMINS` and `PODIUM_MULTI_TENANT` rows above and the multi-tenant operator tenant-list paragraph above; `PODIUM_MULTI_TENANT` is newly documented in the spec although the registry already implements it (`internal/serverboot/serverboot.go:1728`).

## Spec amendment: §13.4 migrations

§13.4 (`spec/13-deployment.md:78`) states that the registry applies its metadata-store schema from the binary on startup, "so a binary upgrade migrates an existing database forward in place, without a separate migration step and without downtime." The multi-tenant operator tenant-list read (§7.3.3) adds no manual step: `applySchema` admits the `*operator-list*` sentinel in the `tenants_org_isolation` policy on `public.tenants` from the binary, updating the existing policy in place on an already-provisioned database (see Store interface additions), because creating, altering, and forcing a policy require only table ownership the registry's `PODIUM_POSTGRES_DSN` role holds. The in-place `CREATE OR REPLACE POLICY` is atomic, so §13.4's no-manual-step and no-downtime guarantee stands unchanged, and this proposal amends no §13.4 sentence.

## Spec amendment: §6.10 error codes

The tenant-management endpoints reuse existing codes for authorization and lookup, and add one code in the existing `registry.*` namespace:

- A request without operator authorization is rejected with `auth.forbidden` (§6.10), the code the per-tenant admin endpoints already return (`pkg/registry/server/admin.go:24`).
- A `PATCH` or `DELETE` naming an unknown tenant is rejected with `registry.tenant_not_found` (§6.10), the code `store.GetTenant` already surfaces through `ErrTenantNotFound` (`pkg/store/store.go:50`).
- A `POST` naming an already-provisioned tenant is not an error; it returns the existing tenant, because provisioning is idempotent (§4.7.1 Provisioning).
- A `POST`, `PATCH`, or `DELETE /v1/admin/tenants` request on a read-only registry (§13.2.1) is rejected with `registry.read_only` (§6.10), the code every other write endpoint returns. The registry checks this after operator authorization and before mutating the store, because `CreateTenant` provisions the org schema (a `CREATE SCHEMA` DDL through `ensureOrg`, `pkg/store/postgres.go:457`) on Postgres and would otherwise fail against a read replica with an opaque database error rather than the structured `registry.read_only`.
- A `/v1/admin/tenants` request on a single-tenant or standalone backend (§13.10), where multi-tenancy is out of scope, is rejected with `registry.tenant_management_unavailable` (§6.10), in the existing `registry.*` namespace.

The new code joins the existing `registry.*` namespace; no new namespace is introduced.

## Spec amendment: §4.7.8 zero-value quota semantics

§4.7.8 (`spec/04-artifact-model.md:860`) lists the per-org limits and the audit-volume daily cap, but defines no meaning for a zero value on any quota field. The runtime create and update endpoints accept a zero-value `Quota`, so the enforcement-time meaning of a zero field must be defined. The appended paragraph below names the user-layer cap as the exception to the zero-value rule, and this proposal makes `max_user_layers` a first-class quota sub-field in the PATCH body, a `podium admin tenant` CLI flag, and a member of the `Quota` struct. The §4.7.8 enumeration at `spec/04-artifact-model.md:862` omits the user-layer cap, so the appended paragraph would name a quota dimension the section never introduces. Amend the enumeration first. Replace:

> Per-org limits, admin-configurable: storage (bytes), search QPS, materialization rate, audit volume.

with:

> Per-org limits, admin-configurable: storage (bytes), search QPS, materialization rate, audit volume, and the per-identity user-defined-layer cap (§7.3.1).

The user-layer cap is the §7.3.1 cap (default 3 user-defined layers per identity, configurable per tenant), surfaced here as a per-org limit so the §4.7.8 zero-value rule covers it. Then add a paragraph to §4.7.8:

> A zero value disables the corresponding budget. A `storage`, `search QPS`, `materialization rate`, or `audit volume` of zero is treated as no limit, so enforcement applies only when an operator sets a positive value. The default user-layer cap is the exception: zero selects the deployment default (3 user-defined layers per identity) and a negative value disables the cap.

This matches the implemented enforcement: storage enforces only when the quota is positive (`pkg/registry/ingest/ingest.go`), the search and materialization rate limiters skip a non-positive rate (`pkg/registry/server/rate_limit.go:94`, `:110`), the audit-volume cap applies only when positive (`internal/serverboot/serverboot.go:1186`), and the user-layer cap reads zero as the default and a negative value as disabled (`pkg/store/store.go:82-85`).

## Open questions

1. **Operator authorization realization.** This proposal seeds the operator role through `PODIUM_OPERATOR_ADMINS` and stores it in a dedicated cross-org operator-grant table. An alternative is a separate operator credential (a token distinct from caller identity) checked at the endpoint, closer to how `migrate-to-standard` runs as a backend operator outside the request-authorization path. The grant model reuses the existing admin-grant audit path and audits uniformly; the separate-credential model decouples operator access from the identity provider. The grant model is proposed; the alternative is noted for review. On the standard backend, the operator list reads `public.tenants` past its FORCE'd policy by setting `podium.org_id` to the `*operator-list*` sentinel that the policy admits, on the registry's existing table-owner connection (§7.3.3, §13.12); this re-scopes the policy rather than bypassing it, so it needs no Postgres `BYPASSRLS` role and no deployment prerequisite, and it works on managed Postgres (§13.1).
2. **Standalone applicability.** Multi-tenancy is a standard-backend capability (§13.1); §13.10 scopes multi-tenancy out of standalone. The tenant-management endpoints are therefore meaningful on a multi-tenant standard backend. On a single-tenant registry (a standalone backend, or a standard backend started without `PODIUM_MULTI_TENANT`) the registry binds every request to its sole tenant and does not consult the organization value (§6.3.1), so a tenant these endpoints create would never resolve. This proposal therefore scopes the tenant-management endpoints to a multi-tenant registry and the registry rejects them when not started in multi-tenant mode, consistent with §6.3.1 and §13.10.
3. **List pagination.** `GET /v1/admin/tenants` returns every tenant. A deployment with a large tenant count may need pagination; this proposal returns the full set and leaves pagination to a follow-up if a deployment's tenant count warrants it.
4. **Hard purge.** Destroying a tenant's data is deferred to a follow-up alongside the §8.5 erasure path, because it is retention-sensitive and not required to close the onboarding gap.

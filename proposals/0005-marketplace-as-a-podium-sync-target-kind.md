# Proposal 0005: Marketplace publishing as a `podium sync` target kind

- Issue: (to be filed)
- Status: Approved (2026-06-25). Converged after 11 adversarial review rounds (15 findings fixed).
- Date: 2026-06-25

## Summary

Proposal 0003 (multi-harness marketplace publishing), applied to the spec in commit `7afeb5c`, added marketplace publishing as a second client surface parallel to `podium sync`: a standalone `podium publish` command and a separate `publish.yaml` config (§7.8), a derived-consumer classification in §2.1 and §2.2, and a glossary "Marketplace output" entry tied to `publish.yaml`. The two surfaces read the same effective view over the same registry HTTP API, render through the same materialization writer into a filesystem directory, and reuse the same selection, lock, and reconciliation helpers. `pkg/publish` imports `pkg/sync`, `pkg/adapter`, and `pkg/materialize`; `PluginFilter.ScopeFilter()` returns a `sync.ScopeFilter` (`pkg/publish/config.go:77`); `render.go` reuses `sync.FetchRecords` (`pkg/publish/render.go:104`), `sync.ReadLock`/`sync.WriteLock` (`pkg/publish/render.go:289,397`), and `sync.Reconcile` (`pkg/publish/render.go:317`); and a marketplace output writes through the same `materialize.Write(workdir, files)` (`pkg/publish/render.go:313`) that `podium sync` writes through (`pkg/sync/sync.go:309`). The operational difference is the output format (the project-files layout versus the marketplace emitters) and a per-output workflow of operator shell commands that clone and push.

This proposal removes the parallel surface. A marketplace becomes a `podium sync` target whose output format is selected by an explicit `kind:` field on the target entry: `kind: workspace` (the default, today's project-files materialization into a local directory the harness reads) or `kind: marketplace` (the marketplace emitters). The `targets:` list in `sync.yaml` holds both kinds. `workflow:` becomes a per-target capability available to either kind. The marketplace-specific fields (the harness set, the git remote and branch, the commit message, the plugins, and the publishing identity) are valid only on a `kind: marketplace` target. `publish.yaml` and the `podium publish` command are removed, and a marketplace target is rendered by `podium sync`. The §2.1 and §2.2 classification of `podium publish` as a separate consumer surface is reverted: `podium sync` is the single materialization consumer, and a marketplace is one kind of target it renders.

The 0003 implementation is complete on this branch. The `podium publish` command is registered in the `cmd/podium/main.go` dispatch table (the `case "publish":` at `cmd/podium/main.go:49-50` and the usage line at `cmd/podium/main.go:147`), implemented in `cmd/podium/publish.go`, and covered by `cmd/podium/publish_test.go` and the end-to-end suite (`test/e2e/publishing_test.go`). `pkg/publish` holds the `publish.yaml` loader (`config.go`), the render pipeline (`render.go`), and the workflow runner (`workflow.go`). Moving the marketplace schema and render path into `pkg/sync` and selecting them by target kind removes a finished, tested CLI surface and reconciles it into the existing `podium sync` consumer.

## Decisions

These are settled design decisions for this proposal. They reverse the separate-surface direction 0003 took, and they are the premises the spec amendments below encode.

1. **A `kind:` discriminant on a target entry.** A `targets:` entry gains a `kind` field with values `workspace` (the default) and `marketplace`. A marketplace is a target, so the `targets:` list holds both kinds, and `kind` declares the output format explicitly rather than leaving the config reader or the operator to infer it from the presence of other fields. `kind` is a new key. It does not reuse the existing `type:` key on a target entry, which is the artifact-type scope filter (`pkg/sync/config.go:72`; the §7.5.2 schema uses `type: [skill, agent]`), so the two coexist without collision.

2. **No `publish.yaml`, and no `podium publish` command.** The marketplace-output configuration moves onto `kind: marketplace` target entries in `sync.yaml`, and `podium sync` renders a marketplace target. The §7.8 `publish.yaml` schema and the `podium publish` definition are superseded by the §7.5.2 `sync.yaml` schema and the `podium sync` command. The standalone command is built and tested on this branch (`cmd/podium/publish.go`, the `case "publish":` dispatch at `cmd/podium/main.go:49-50`, and `cmd/podium/publish_test.go`), so this change removes a finished surface, and the removal steps below delete its files and dispatch entry.

3. **`workflow:` is a per-target capability available to either kind.** The `prepare` and `publish` command lists (each an argv `run:` or a shell `sh:` command, with `skip_if_no_changes`, `continue_on_error`, `timeout`, and the per-phase `prepare_on_error`/`publish_on_error` cleanup, `pkg/publish/config.go:89-94`) attach to a target. A `kind: marketplace` target uses `workflow.publish` to clone and push the rendered repository. A `kind: workspace` target may run post-sync lifecycle commands such as a commit, a formatter, or a reload. 0003 carried the workflow on the marketplace output alone; this proposal makes it a target field.

4. **Podium always renders to a folder, and the git push lives in the target's `workflow`.** Podium never holds a git push credential and never pushes (0003 placed the push in operator-configured commands). A marketplace target's destination is the `target:` directory, the same field a workspace target writes to; for a marketplace target it is the working directory the `prepare` phase places a checkout into and the `publish` phase takes to the remote. There is no directory-versus-remote destination type inside Podium.

5. **`podium sync` is the single materialization consumer, and a marketplace is one kind of target.** 0003's §2.1 and §2.2 classify `podium publish` as a derived, served output distinct from `podium sync` on consumer role (`spec/02-architecture.md:7,50,97`). That distinction does not hold once publishing renders to a folder through the same writer and reads the effective view over the same HTTP API. The clone-before-render and push-after-render lifecycle is the marketplace target's `workflow` (Decision 3), so it is a property of the target rather than of a separate consumer. This proposal reverts the §2.1 and §2.2 classification so the consumer enumeration names `podium sync` once, rendering workspace and marketplace output kinds. The §1.3 closed list of runtime consumer paths (the SDKs, the MCP server, and `podium sync`) already omits `podium publish`, so it stays consistent and is left unchanged.

6. **The publishing identity is a field of a `kind: marketplace` target, inherited from an optional `defaults.identity`.** The §4.6 effective-view principal governs which visibility-filtered content reaches a public marketplace, and §7.8 names it a security-relevant setting. `publish.yaml` carried it in `defaults.identity` and copied it into the resolved output (`pkg/publish/config.go:39,307`); this proposal carries it on the marketplace target so the security property survives the move. The declared identity is only the documentary principal name: `RenderOptions.Identity` is passed through but not read by `Render` (`pkg/publish/render.go:92-95`), and the registry credential carries the publishing identity. The effective view is fetched under that credential, so the security property is preserved by resolving and passing the correct credential rather than by an identity-versus-token comparison. The marketplace dispatch resolves and passes the credential as detailed in the dispatch section. A `kind: workspace` target renders under the caller's own identity and carries no identity field, and the existing `sync.yaml` `defaults:` block (`pkg/sync/config.go:32-48`) is the workspace-materialization default block, so the publishing identity is a marketplace-scoped default rather than a workspace one.

7. **The marketplace emitter mechanism is unchanged.** `adapter.EmitterForHarness`, the `MarketplaceEmitter` contract, and the `PluginDescriptor` (`pkg/adapter/adapter.go`) take a harness ID, a `Source`, and a marketplace name, and carry no dependency on the config surface. The §9.1 `HarnessAdapter` plugin-descriptor amendment 0003 applied stays as written. Only the trigger that selects the marketplace emitters versus the project-files materialization changes: it runs for a `kind: marketplace` target.

8. **The MCP server and bridge parse `sync.yaml` but never execute a `workflow`.** `cmd/podium-mcp` reads `sync.yaml` for the defaults, the profiles, and the version pin (`cmd/podium-mcp/cliconfig.go:223`) and for the overlay watch (`cmd/podium-mcp/overlay_watch.go`), and it never iterates `targets:` or runs a command. Because `workflow` is now a field on any target, `sync.yaml` is execution-bearing, so this proposal makes it an explicit invariant that a `workflow` executes only on the `podium sync` CLI path and never in the MCP server or the bridge, so the always-on multi-tenant server cannot become a command-execution path.

## Spec amendment: §7.5.2 the `kind:` discriminant and marketplace target fields

Amend the §7.5.2 **Schema** block so a `targets:` entry carries a `kind:` field and, when `kind: marketplace`, the marketplace-output fields. Add the marketplace target to the schema example after the existing `targets:` entries:

```yaml
targets:
  - id: claude-code
    kind: workspace # the default; materializes the project-files layout
    harness: claude-code
    target: ~/.claude/
    profile: project-default

  - id: acme-agents
    kind: marketplace # renders the marketplace emitters (§7.8)
    harnesses: [claude-code, codex, cursor]
    target: ./build/acme-agents # the working directory the checkout lands in
    git:
      remote: git@github.com:acme/agent-marketplace.git
      branch: main
    commit_message: "Sync Podium catalog ({{.ChangedCount}}) {{.Timestamp}}"
    identity: publisher@acme.com # or inherited from defaults.identity
    plugins:
      - name: finance-pack
        include: ["finance/**"]
    workflow:
      prepare:
        - run: ["git", "clone", "$PODIUM_GIT_REMOTE", "$PODIUM_WORKDIR"]
      publish:
        - run: ["git", "-C", "$PODIUM_WORKDIR", "add", "-A"]
        - run: ["git", "-C", "$PODIUM_WORKDIR", "commit", "-m", "$PODIUM_COMMIT_MESSAGE"]
          skip_if_no_changes: true
        - run: ["git", "-C", "$PODIUM_WORKDIR", "push", "origin", "$PODIUM_GIT_BRANCH"]
```

Add the behavioral description to §7.5.2:

> A `targets:` entry carries a `kind:` field that selects its output format. `kind: workspace` is the default and materializes the project-files layout into the target directory, the layout the harness reads directly. `kind: marketplace` renders the harness's git-repo distribution layout (§7.8) into the target directory as a working checkout. A `kind: workspace` entry carries the harness, the target directory, and the scope fields (`profile`, `include`, `exclude`, and `type`). A `kind: marketplace` entry carries a harness set (`harnesses`), a git remote and branch, a commit message, a plugin list, and a publishing identity. Both kinds may carry a `workflow:` of `prepare` and `publish` command lists. The marketplace fields are rejected on a `kind: workspace` entry, and the workspace scope fields, the watch mode (§7.5.4), and the ephemeral overrides (§7.5.5) are rejected on a `kind: marketplace` entry. The publishing identity is the §4.6 effective-view principal whose visibility defines what reaches the marketplace, and a marketplace target inherits it from an optional `defaults.identity` when it sets none of its own. A configuration that declares only `kind: workspace` targets, or omits `kind` entirely, behaves exactly as before, because `kind` defaults to `workspace`.

Add the `workflow` execution boundary to §7.5.2:

> A `workflow:` runs only when `podium sync` materializes a target. The MCP server and the in-process bridge read `sync.yaml` for the registry, the profiles, and the version pin, and they never execute a target's `workflow`. A `workflow` command therefore runs with the privileges of the `podium sync` process and never inside the registry server.

Anchor: the **Schema** addition extends the `targets:` example that ends at `spec/07-external-integration.md:272`, and the behavioral description and the execution boundary follow it in §7.5.2. The `kind` field is added to `TargetEntry`, and the marketplace fields reuse the marketplace-output schema relocated from §7.8 into §7.5.2 by the §7.8 amendment below.

## Proposed solution

### Add the `kind` discriminant and the marketplace fields to `TargetEntry`

Add a `Kind string` field to `TargetEntry` (`pkg/sync/config.go:64-73`) with the YAML tag `kind,omitempty`, defaulting to `workspace` when empty. Add the marketplace fields to `TargetEntry`: the harness set, the git remote and branch, the commit message, the plugins, the publishing identity, and the workflow. Move the `MarketplaceOutput` component types (`GitRemote`, `PluginFilter`, `Workflow`, `Command`, and `Duration`), the resolved `ResolvedOutput` type (`pkg/publish/config.go:265`), the `validateOutput` function (`pkg/publish/config.go:346`), and the `ErrConfigInvalid` error var (`pkg/publish/config.go:23`) from `pkg/publish/config.go` into `pkg/sync` so they live beside the `ScopeFilter`, lock, and reconciliation machinery they already reuse, and `PluginFilter.ScopeFilter()` becomes a same-package call (`pkg/publish/config.go:77`). `ResolvedOutput` is the marketplace payload the re-homed workflow runner consumes through `RunOptions.Output` (`pkg/publish/workflow.go:49`); the re-home section below details that dependency. Add an optional `Identity string` field to the `sync.yaml` `Defaults` block so a marketplace target inherits the publishing identity, keeping the §4.6 effective-view rule enforceable from `sync.yaml` (Decision 6).

Carry the new fields through the multi-target plan. `MultiTargetPlan` (`pkg/sync/resolve.go:311-318`) is the resolved per-entry record the `runMultiTargetSync` loop iterates, and `PlanMultiTarget` (`pkg/sync/resolve.go:330`) flattens each `TargetEntry` into one. Add a `Kind` field and the marketplace payload (the harness set, the git remote and branch, the commit message, the plugins, the resolved publishing identity, and the workflow) to `MultiTargetPlan`, and populate them in `PlanMultiTarget` (`pkg/sync/resolve.go:350`) so the per-`kind` dispatch below has the marketplace fields to read. For a `kind: marketplace` entry `PlanMultiTarget` resolves the marketplace payload and skips the workspace `targetScope` resolution (`pkg/sync/resolve.go:365`), which otherwise returns an empty, non-error scope and would dispatch the entry as a `none` workspace materialization.

Add config validation: a `kind: workspace` target rejects the marketplace fields (the harness set, the git remote and branch, the commit message, the plugins, and the publishing identity), and a `kind: marketplace` target rejects the workspace scope fields, the watch mode, and the ephemeral overrides. `workflow` is a universal per-target field (Decision 3), so it is accepted on both kinds and is not in either reject set. A marketplace harness set that names a non-publish harness (OpenCode or `none`) is rejected, the §7.8 rule the relocated schema carries forward. The kind validation runs in `PlanMultiTarget` at the start of a `podium sync --config` run, and `podium sync --check --config <path>` performs the same validation without materializing either kind, because the dispatch section routes a `--check` run that carries `--config` through the multi-target path and runs each plan's no-side-effect path (a marketplace plan through `RunMarketplace` with `opts.Check`, a workspace plan through `sync.Run` with `DryRun` set), so no target tree and no `<target>/.podium/sync.lock` is written.

### Re-home the render pipeline and the workflow runner into `pkg/sync`

Move the render pipeline (`Render`, the emitter selection, the plugin assignment, the reconcile and change-set helpers, and the `RenderOptions`/`RenderResult` types from `pkg/publish/render.go`) and the workflow runner (`pkg/publish/workflow.go` and its coverage `pkg/publish/workflow_test.go`) into `pkg/sync`. The `sync.FetchRecords` (`pkg/publish/render.go:104`), `sync.ReadLock`/`sync.WriteLock` (`pkg/publish/render.go:289,397`), and `sync.Reconcile` (`pkg/publish/render.go:317`) calls the render makes become same-package calls. The `adapter.PodiumOwnedKey` tag the reconciliation relies on stays in `pkg/adapter` as a cross-package reference (`pkg/adapter/adapter.go:58`). The runner calls `Render(ctx, RenderOptions{...})` (`pkg/publish/workflow.go:145`) and uses the render result types, so it moves with them. The workflow runner also depends on the `ResolvedOutput` type (`pkg/publish/config.go:265`): `RunOptions.Output` is a `ResolvedOutput` (`pkg/publish/workflow.go:49`), and `runPipeline` and `baseVars` take one (`pkg/publish/workflow.go:130,316`). `ResolvedOutput` is defined in `config.go`, which the removal section below deletes with the package, and after the planned removals the workflow runner is its only remaining production user. `ResolvedOutput` therefore moves into `pkg/sync` alongside the workflow runner so the re-homed runner compiles. Its fields (`ID`, `Registry`, `Identity`, `Git`, `Harnesses`, `CommitMessage`, `Plugins`, and `Workflow`) are the resolved marketplace payload the multi-target plan carries (the same fields the dispatch section adds to `MultiTargetPlan`), so the dispatch builds a `ResolvedOutput` from a `kind: marketplace` plan and passes it as `RunOptions.Output`. The runner's `--check` path also hard-depends on two further `config.go` symbols that the removal deletes with the package: the package function `validateOutput` (called at `pkg/publish/workflow.go:109` inside the `if opts.Check {` branch) and the package error var `ErrConfigInvalid` (used at `pkg/publish/workflow.go:349,353,434,436`). `pkg/sync` has no namesake for either: `pkg/sync/config.go:18` defines only `ErrConfigNotFound`, and its only `config.invalid` string is `config.invalid_min_version` (`pkg/sync/config.go:250`). Both `validateOutput` (`pkg/publish/config.go:346`) and `ErrConfigInvalid` (`pkg/publish/config.go:23`) therefore move into `pkg/sync` alongside the runner so it compiles, mirroring the `ResolvedOutput` move. `validateOutput`'s own dependencies resolve once the runner is same-package: `Workflow.commands()` (`pkg/publish/config.go:111`) moves with `Workflow`, `Command.validate()` (`pkg/publish/workflow.go:431`) moves with the runner, `sync.ValidateGlob` (`pkg/sync/scope.go:18`) becomes a same-package call, and `adapter.EmitterForHarness` stays a cross-package reference. `ErrConfigInvalid` does not collide with any `pkg/sync` symbol, so it moves without a rename. `validateOutput` performs the §7.8 publish-target enforcement on the `--check` path (it rejects a non-publish harness via `adapter.EmitterForHarness`, a malformed plugin glob, and a command declaring neither `run:` nor `sh:`), which the `PlanMultiTarget` kind validation (line 145) does not subsume, so the moved runner keeps calling `validateOutput(out)`. `pkg/sync` already imports `pkg/adapter` and `pkg/materialize` and does not import `pkg/publish`, so the move introduces no import cycle. `pkg/sync` already exports its own `func Run(opts Options) (*Result, error)` (`pkg/sync/sync.go:180`), so the workflow runner's `func Run(ctx context.Context, opts RunOptions) (*RunResult, error)` (`pkg/publish/workflow.go:93`) is renamed to `RunMarketplace` when it moves into `pkg/sync`, to avoid a duplicate `Run` declaration. The render result types (`RunResult`, `RenderOptions`, `RenderResult`, and `Render`) have no `pkg/sync` namesakes, and `ResolvedOutput` has none either, so the bare `Run` function is the only moved symbol that needs a rename.

## Relationship to proposals 0003 and 0004

This proposal supersedes 0003's separate-surface framing. 0003 applied §7.8, the §2.1 and §2.2 `podium publish` classification, the glossary "Marketplace output" entry, and the §9.1 plugin-descriptor note, and it built `pkg/publish` (the loader, the render pipeline, and the workflow runner) together with the `cmd/podium/publish.go` command and its `case "publish":` dispatch (`cmd/podium/main.go:49-50`). The marketplace emitters, the `HarnessAdapter` plugin descriptor, the repository layout, the publish-target set, and the reconciliation guarantee that 0003 introduced are carried forward unchanged.

This proposal is disjoint from 0004 (webhook hardening). 0004 amends §7.3.2 (receiver authorization, the SSRF policy, the per-receiver debounce window, and the batch delivery body), §13.12 (`PODIUM_WEBHOOK_ALLOWED_TARGETS`), and §6.10 (which reuses existing codes), and it changes the webhook subsystem code (`pkg/registry/server/webhooks.go`, `pkg/webhook/webhook.go`, and `pkg/registry/server/identity_verify.go`). This proposal does not amend §7.3.2 and does not touch the webhook subsystem. Keeping §7.8 as its section number leaves the §7.3.2 cross-reference to §7.8 (`spec/07-external-integration.md:123`) valid without editing §7.3.2, and the §7.8 references to proposal 0004 are preserved verbatim, so the marketplace trigger continues to read the existing `layer.ingested` event and 0004's debounce window without redefining either. 0004 applies before this proposal, and the two share no spec section and no code path.

## Open questions

1. **Trust gate for a target `workflow`.** Because `workflow` is a field on any target, a committed project-shared `sync.yaml` can carry shell commands that `podium sync` executes, the npm-postinstall or direnv risk class. The Decision 8 invariant keeps the MCP server from executing them, but the CLI still does. Whether to add a trust gate (a direnv-style per-file allow, a restriction that honors a `workflow` only from the gitignored `sync.local.yaml` scope, or a `--allow-commands` flag) belongs to the reviewer. The three-scope model (`~/.podium/`, `<workspace>/.podium/sync.yaml`, and `<workspace>/.podium/sync.local.yaml`, §7.5.2) is where a scope restriction would draw the boundary.
2. **Single-target marketplace invocation.** A marketplace target is reachable through `podium sync --config <path>` because the git remote and the harness set have no single-target CLI-flag analog. Whether a `podium sync --target <id>` selector is added to render one marketplace target from a multi-target config, for parity with the dropped `podium publish --output`, is open.
3. **Default `kind`.** `kind` defaults to `workspace` so every existing `sync.yaml` parses unchanged. Whether `kind` is required on a new target entry for explicitness, or stays optional with the `workspace` default, is open.
4. **Workflow override granularity.** A marketplace target carries its full `workflow`. Whether a `defaults.workflow` block is inherited and overridable per target (as `publish.yaml` allowed) or each marketplace target declares its own is open; inheritance from `defaults.workflow` is the smaller config for the common case.
